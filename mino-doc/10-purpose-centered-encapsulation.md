# 目的を中心にカプセル化する

## このノウハウの核

カプセル化は、単にfieldをprivateにすることではありません。特定の目的に強く関係するデータと振る舞いを一つの責務所有者へ集め、不正な変更や目的外の操作を外部へ許さず、正しい操作だけを公開することです。

関心の分離とカプセル化は対になります。

- 強く関係するものを、目的単位でカプセル化する。
- 関係の弱い、目的の異なるものを分離する。
- 一つの目的を変更するとき、別の目的を忘れられる構造にする。

Skill は、凝集度や結合度の数値だけでなく、目的、所有状態、公開操作、変更影響からカプセル化を評価します。

## 解決する問題

- public fieldやsetterにより、どこからでも状態を変更できる。
- データと、それを扱うロジックが別々のUtilやServiceへ散らばる。
- 巨大クラスの内部ではメンバが実質グローバル変数・グローバル関数になる。
- インスタンスメソッドが自身の状態を使わず、他オブジェクトを操作する。
- 一つのクラスが複数のアクター・ユースケースのデータを持つ。
- 正しい操作順や不変条件が、呼び出し側の注意に依存する。
- 「クラスを増やしたくない」という理由で、既存クラスへ責務を追加し続ける。

## 必要な入力

- 対象クラス・モジュールと全利用箇所。
- アクター、目的、ユースケース。
- 状態、状態を読む・変えるロジック、制約。
- public API、setter、field access、直接DB更新。
- 変更履歴と、同時に変更される要素。
- 目的ごとの必要データと不要データ。
- 現在のテストと不変条件。

## 判断規則

### 1. カプセルの単位は目的で決める

「同じ商品データだから」「同じテーブルだから」という理由だけでまとめません。目的が違い、ルールや変更理由が違うなら、別のカプセル候補です。

```text
注文目的: 注文品 + 数量制約 + 価格確定
配送目的: 配送品 + 宛先 + 出荷状態
```

### 2. データの所有者へ強く関与するロジックを移す

次を満たすロジックは、原則としてその状態を所有するモデルの候補です。

- 値の正しさを判断する。
- 状態を変える。
- 状態から業務上の結果を計算する。
- 状態遷移を許可・拒否する。

UtilやServiceに置かれている場合、呼び出し側が内部構造を知りすぎていないか確認します。

### 3. 正しい操作だけを公開する

- `setStatus(APPROVED)` より `approve(by, at)`。
- `setAmount(x)` より `increaseBy(delta)`、`applyDiscount(policy)`。
- collection getterより、`addLine`、`removeLine`、`total`。

公開APIは業務上の意図と事前条件を表し、内部表現を直接操作させません。

### 4. インスタンスメソッドは自身の状態に責任を持つ

対象オブジェクトの状態を使わず、引数で渡された別オブジェクトだけを操作するメソッドは、責務の置き場所が不自然な可能性があります。メソッドを所有状態の側へ移す、Policyとして分離する、または純粋関数にすることを検討します。

### 5. 生成時から不変条件を守る

不正な未初期化状態を作ってsetterで組み立てるのではなく、Constructor / Factoryで必須情報と制約を保証します。状態変更後も不変条件を守れる操作だけを持たせます。

### 6. 情報隠蔽と観測可能性を両立する

内部状態を隠しても、業務上必要な問いには答えられるようにします。ただし、内部コレクションや可変参照をそのまま返さず、意味のあるquery、immutable view、snapshotを使います。

### 7. 小ささを目的化しない

クラスを分割した結果、一つの不変条件を守るために多数のオブジェクトを巡回するなら、分割しすぎです。目的、変更理由、整合性を一緒に考えます。

## 実行手順

1. 対象のpublic API、状態、ロジック、利用箇所を調査する。
2. 各要素に、どのアクター・目的・ユースケースのためかを付与する。
3. 目的が異なる要素をグループ分けする。
4. 各グループで、状態の正の所有者と不変条件を決める。
5. 所有者外へ散らばる判断・計算・更新を特定する。
6. 正しい業務操作として公開APIを再設計する。
7. public setter、可変参照、直接更新の迂回路を閉じる。
8. 目的の異なるグループを別クラス・モジュールへ分ける。
9. Characterization testと不変条件テストを作る。
10. 呼び出し側を段階的に新APIへ移し、旧APIの利用がゼロになったら削除する。

## 完了条件

- [ ] 各カプセルに一貫した目的がある。
- [ ] 強く関係する状態、判断、計算、更新が権威ある所有者へ集まっている。
- [ ] public APIが業務上の意図を表し、任意setterに依存していない。
- [ ] 不正状態を生成・変更する迂回路が閉じられている。
- [ ] インスタンスメソッドが自身の状態または責務に関係している。
- [ ] 目的外のデータと操作が対象外として分離されている。
- [ ] 一つの不変条件が過度な分割で守れなくなっていない。
- [ ] 旧APIから新APIへの移行と回帰テストがある。

## 失敗パターン

- getter/setterを生成し、fieldをprivateにしただけで完了とする。
- すべてのロジックをModelへ押し込み、技術関心や別目的まで混ぜる。
- データとロジックを分けること自体を疎結合と呼び、所有者を失う。
- クラス追加を悪とみなし、巨大Serviceへ処理を足し続ける。
- 小さなクラスを大量に作り、意味のある操作が見えなくなる。
- mutable collectionをgetterで返し、外部から自由に変更できる。
- 新しいメソッド名だけを付け、内部ではsetterを公開したままにする。

## エージェント向けプロンプト骨子

```text
対象クラスまたはモジュールを、目的中心のカプセル化の観点で分析してください。

1. 状態、ロジック、public APIごとに、アクター、目的、ユースケースを特定する。
2. 目的が異なる要素の混在を指摘する。
3. 状態の正の所有者と不変条件を定義する。
4. 所有者外へ散らばる判断、計算、更新を移す候補を示す。
5. setterや内部表現の公開を、意味のあるcommand/queryへ置き換える。
6. インスタンスメソッドが自身の状態を使わない箇所を検査する。
7. 分割しすぎで整合性を失わない境界を提案する。
8. 挙動維持テストと段階移行を示す。
```
