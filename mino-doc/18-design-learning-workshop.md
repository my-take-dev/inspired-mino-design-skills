# プロダクションコードで設計力を育てるワークショップ

## このノウハウの核

設計知識は、講義や読書だけでは実務で使える技能になりにくいものです。変更容易性の高い設計を身につけるには、実際のコードから問題を発見し、問題を言語化し、再設計し、その理由と効果を説明するところまで自分で行う必要があります。

このSkillは、プロダクションコードを題材に、次の能力を反復可能なワークショップで育成します。

- 問題のあるコードを自力で認識する。
- 問題を、設計原則と変更シナリオに基づいて説明する。
- 泥臭く複雑なコードを、目的に沿った理想構造へ再設計する。
- 変更容易性の高い構造要件を満たす。
- 何が問題で、なぜその設計で解決できるのかを筋道立てて説明する。

重要なのは、講師やAIが正解を先に与えないことです。学習者が手と口を動かし、認識、言語化、設計、説明の各工程を自分で行います。

## 解決する問題

次の症状があるときに使います。

- 設計本を読んだが、実コードで問題箇所を見つけられない。
- 「なんとなく汚い」は言えるが、何の品質がどう悪いか説明できない。
- リファクタリング案を模倣できても、別のコードへ応用できない。
- フレームワークや実装技術へ注意が偏り、解くべき設計問題を見失う。
- チーム内で設計レビューの語彙と判断基準が揃っていない。
- AIの改善案を、良いか悪いか人間が評価できない。
- 勉強会が講義中心で、受講者の理解度を観測できない。

緊急障害対応中のコード、機密性が高く共有できないコード、学習者が安全に扱えない本番データを含む題材には、そのまま適用しません。

## 学習成果の定義

「理解した」「受講した」ではなく、次を観測できたときに習得とみなします。

1. 対象テーマに関する問題コードを、自力で発見できる。
2. 観測事実、破られた設計要件、変更時の悪影響を説明できる。
3. 元コードの偶然の形に引きずられず、理想構造を再設計できる。
4. 再設計後が、テーマの設計要件と維持すべき契約を満たす。
5. 採用した責務分割、名前、境界の理由を説明できる。
6. フィードバックを受けて、問題認識または設計を修正できる。

## 推奨する学習テーマ

基礎Skillは一度に一つだけ扱います。

1. カプセル化
2. 関心の分離
3. 単一責任原則
4. interface設計
5. 目的駆動の名前設計

チームの成熟度に応じて、契約による設計、ドメインモデル完全性、境界付けられたコンテキスト、技術的負債分析を後続テーマとして追加できます。

## 必要な入力

- 今回の学習テーマと、その設計要件。
- 対象リポジトリまたは匿名化したプロダクションコード。
- 維持すべき外部挙動、既存テスト、業務ルール。
- 学習者の経験、対象言語、1回あたりの時間。
- コード共有、画面共有、機密情報に関するルール。
- 発表とフィードバックを行う4〜5人以内の参加者。
- ファシリテーターと、必要に応じたドメイン知識提供者。

## ワークショップの安全境界

- 実習はscratch branch、sandbox、コピー、または新規ファイルで行う。
- 本番ブランチへ自動commit、push、deployしない。
- 秘密情報、個人情報、本番データを教材へ含めない。
- 学習用の再設計を、そのまま本番採用しない。別途レビュー、テスト、移行設計を行う。
- AIはヒント、論点整理、rubric評価に使えるが、学習者の問題発見と説明を代行しない。
- 人ではなく、コード、説明、根拠を批評する。

## 判断規則

### 1. 一回につき一つの設計テーマへ集中する

複数原則を同時に採点すると、学習者が何を認識できていないのか判別できません。主テーマ以外の問題は`parking_lot`へ退避します。

### 2. 問題発見は各自で行う

ペアやモブだけで問題を探すと、詳しい人へ依存し、自分で問題を認知する練習が不足します。問題候補の議論は推奨しますが、最終的な選定と説明メモは各自で作ります。

### 3. コードを書く前に問題を言語化する

再設計を始める前に、最低限次を書きます。

- どのコードが問題か。
- 何が観測できるか。
- どの設計要件または品質特性に反するか。
- どの変更シナリオで困るか。
- 理想構造では何がtrueになるべきか。

問題認識が不正確なまま設計すると、手段も不正確になります。

### 4. 学習用再設計はゼロから書く

既存コードをその場で少しずつリファクタリングすると、互換性、周辺依存、移行手順へ注意が分散します。学習時は新規ファイルへ理想構造を書き、テーマの設計要件に集中します。

本番適用時は、別途`14-legacy-refactoring-by-purpose.md`などを使って安全な移行を計画します。

### 5. 設計後も理由を言語化する

完成コードだけでは習得度を評価できません。次を説明できることを必須にします。

- 何をどの責務へ移したか。
- なぜその境界、名前、interfaceにしたか。
- 元の問題がどのように解消したか。
- 維持した契約と、残った問題は何か。

### 6. 理想構造を先に共有するが、正解コードは与えない

学習者が技術的負債を認識するには、変更容易性の高い構造要件を知る必要があります。一方、完成コードを先に見せると模倣学習になります。共有するのは原則、判定基準、短い例までにします。

## 標準セッション構成

推奨は、1テーマにつき予習1時間と実習2時間です。5テーマなら合計15時間です。人数は発表と質疑応答が成立する4〜5人以内を基本とします。

### Step 1: 予習

**目安: 60分、個人**

- 対象テーマの教材を読む。
- 良い構造と悪い構造の違いを自分の言葉でまとめる。
- 分からない用語、反例、疑問を記録する。

成果物: `preparation_notes`

### Step 2: 概要確認と目線合わせ

**目安: 10〜15分、全体**

- テーマの目的、設計要件、対象外を確認する。
- 学習成果とrubricを共有する。
- 短い例で判定基準を揃える。

成果物: `shared_rubric`

### Step 3: 問題のあるコードを探す

**目安: 20〜25分、個人**

- 普段扱うコードから、テーマを適用できそうな箇所を探す。
- 候補ごとに変更シナリオを一つ考える。
- 最も学習価値が高く、安全に扱える箇所を選ぶ。

成果物: `problem_candidate`

### Step 4: 何が問題かをメモに起こす

**目安: 5〜10分、個人**

```text
対象:
観測事実:
破られた設計要件:
変更時の悪影響:
理想状態:
```

成果物: `problem_explanation`

### Step 5: 問題コードを説明する

**目安: 15〜20分、全体**

- 一人1〜2分で画面共有して説明する。
- 他の参加者は、事実と解釈の混同、テーマずれ、根拠不足を質問する。
- 必要なら問題文を修正する。

成果物: `reviewed_problem_statement`

### Step 6: 再設計して実装する

**目安: 20〜25分、個人**

- 新規ファイルまたはsandboxへ理想構造を書く。
- 元コードの公開契約と必要な業務ルールを維持する。
- テーマの設計要件をコード上で満たす。
- 完成度より、責務、境界、名前の明確さを優先する。

成果物: `redesign_candidate`

### Step 7: どう設計したかをメモに起こす

**目安: 5〜10分、個人**

```text
採用した構造:
責務の移動:
名前と境界の理由:
元の問題への効果:
維持した契約:
残課題:
```

成果物: `design_rationale`

### Step 8: 成果発表

**目安: 20〜25分、全体**

- 一人2〜3分で再設計と理由を説明する。
- チームで質疑応答とフィードバックを行う。
- rubricに基づき、問題認識、構造、説明を分けて評価する。
- 本番採用候補は、ワークショップ外の正式な改善タスクへ移す。

成果物: `reviewed_redesign`、`learning_feedback`

## ファシリテーターの役割

- テーマ、時間、対象外を守る。
- 正解を教える前に、観測事実と変更シナリオを質問する。
- フレームワークや文法の議論が主目的を奪ったら戻す。
- 発言量の偏りを抑え、全員が説明する機会を作る。
- 問題認識が誤っている場合、設計へ進ませず修正する。
- 参加者のコードを人格評価へ結び付けない。
- 未解決の実務課題を、owner付きのfollow-upへ記録する。

AIをファシリテーター補助として使う場合も、学習者に代わって問題文や再設計を完成させません。

## 評価rubric

各項目を0〜3で評価します。

- `0`: 未実施、またはテーマと無関係。
- `1`: 主張はあるが根拠や構造が不十分。
- `2`: 概ね正しいが、一部の責務、契約、説明に欠落がある。
- `3`: 観測、設計要件、再設計、説明が一貫している。

| Dimension | 3点の状態 |
|---|---|
| Problem recognition | 対象テーマの問題を自力で発見し、事実で説明できる |
| Change scenario | どの変更で何が困るか具体的に示せる |
| Design requirement | 破られた要件と理想状態を説明できる |
| Redesign | 目的に沿った責務、境界、名前で再構成されている |
| Contract preservation | 維持すべき振る舞いと制約が失われていない |
| Rationale | なぜその設計で問題が解けるか筋道立てて説明できる |
| Feedback response | 指摘を理解し、問題文または設計を修正できる |

総点だけでなく、`Problem recognition`と`Rationale`をhard gateにします。両方が2未満なら、そのテーマを習得済みとしません。

## 完了条件

- [ ] 一回の主テーマが一つに絞られている。
- [ ] 学習者が問題コードを自力で発見している。
- [ ] 再設計前に問題が文章化され、レビューされている。
- [ ] 学習用実装が本番コードから安全に隔離されている。
- [ ] 再設計がテーマの構造要件と維持契約を満たしている。
- [ ] 学習者が設計理由と問題への効果を説明している。
- [ ] 全参加者が発表とフィードバックに参加している。
- [ ] rubricが記録され、習得、再実施、未完了を判定できる。
- [ ] 本番適用候補が、正式なレビューと移行計画なしに反映されていない。

## 失敗パターン

- 講師が長時間説明し、学習者が手を動かさない。
- 玩具的なサンプルだけを使い、泥臭い実コードへ応用できない。
- 問題発見をチームの詳しい人やAIへ任せる。
- 問題を説明せず、すぐコードを書き換える。
- 一度に多数の設計原則を評価し、学習課題が曖昧になる。
- 既存コードの細部をそのまま直し、理想構造を考えない。
- 完成コードだけを評価し、理由や変更シナリオを確認しない。
- ワークショップ成果を、テストや移行設計なしで本番へ入れる。
- 参加者を比較・順位付けし、質問しにくい雰囲気を作る。

## エージェント向けプロンプト骨子

```text
プロダクションコードを題材に、設計学習ワークショップを設計してください。

1. 一回につき一つの設計テーマと、観測可能な学習成果を定義する。
2. 予習、目線合わせ、個人での問題発見、問題言語化、説明、ゼロからの再設計、設計理由の言語化、成果発表の8段階にする。
3. AIや講師が学習者の問題発見と説明を代行しないようにする。
4. 本番コードへの書き込み、秘密情報、無審査の採用を禁止する。
5. 問題認識、変更シナリオ、構造要件、再設計、契約維持、説明、feedback対応をrubricで評価する。
6. 本番採用候補は、別の改善タスクとして移行・テスト・レビューを要求する。
```
