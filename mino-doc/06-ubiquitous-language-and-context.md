# 文脈に基づくユビキタス言語を設計する

## このノウハウの核

言葉の意味は、単語だけでは一意に決まりません。誰が、何の目的で、どの状況にいて、どのルールの下で使うかによって変わります。開発者が自分の経験や一般的定義だけで解釈すると、要求、モデル、実装が本来の意図からずれます。

Skill は、用語を次の四要素とセットで扱います。

1. **アクター**: 言葉を使う、または影響を受ける人・システム。
2. **目的**: そのアクターが実現したい状態。
3. **文脈**: 背景、状況、業務段階、前後関係。
4. **ルール**: その文脈内で成立する制約、判断、方針。

ユビキタス言語は単なる用語集ではなく、特定の境界付けられたコンテキスト内で、目的、意味、ルールを揃えるための共通言語です。

## 解決する問題

- 「ユーザー」「商品」「類似」「適当」など、一般語の解釈が人によって違う。
- 要求どおりの文字列を実装したが、期待する価値を生まない。
- 同じクラス名が複数の業務役割を背負う。
- ドメインエキスパートと開発者の会話が噛み合わない。
- AIが曖昧な語を一般的な意味で補完し、別の機能を作る。
- 名前変更が可読性の改善だけに留まり、構造や境界を変えない。

## 必要な入力

- 実際の業務会話、仕様、問い合わせ、操作手順。
- アクター、目的、利用場面。
- 例、反例、境界値、例外ケース。
- 現在のコード上の名前、DBカラム、API、画面文言。
- 同じ語を使う部署・コンテキスト一覧。
- 用語に紐づく判断、状態遷移、制約。

辞書や一般的な業界定義は参考にできますが、対象業務での用法を上書きしません。

## 判断規則

### 1. 自分の文脈を事実として投影しない

人間もAIも、既知の枠組みに当てはめて意味を推測します。推測は仮説として明示し、次を確認します。

- この言葉を発したのは誰か。
- その人は何を達成したいのか。
- 発言の前後に何が起きていたか。
- 成功例と失敗例は何か。
- この意味が適用されない場面はどこか。

### 2. 抽象的な共通語より、目的が分かる役割名を優先する

一つの「商品」を全場面で使うのではなく、文脈に応じて「注文品」「配送品」「在庫品」のように役割を表します。一つの「ユーザー」が個人顧客、法人担当者、見込み客、問い合わせ者を表すなら、モデルと名前を分けます。

名前は次を伝える必要があります。

- 誰にとってのものか。
- 何の目的で使うものか。
- どの範囲で意味が有効か。
- どの操作・判断を担うか。

### 3. 用語とモデルを往復させる

ユビキタス言語は文書だけに置かず、クラス、メソッド、イベント、テスト、APIへ反映します。コードにない用語は設計で使われていない可能性があり、コードだけにある用語は業務との接続が失われている可能性があります。

### 4. 同じ語の意味が違うなら境界を分ける

同じ名称を無理に統一定義しません。意味、ルール、ライフサイクルが違う場合は、コンテキストごとの定義を持ち、境界で翻訳します。

各meaningにはstableなterm IDを付けます。曖昧語と翻訳は、アクター、目的、ルールを重複して持たず、canonicalなterm recordをIDで参照します。

### 5. 定義より用法と意図を確認する

「この単語の正しい定義は何か」だけでなく、次を聞きます。

- この言葉を使って、何を判断したいのか。
- どんな操作を許可・禁止したいのか。
- どの事実が変わると、この呼び方ではなくなるのか。
- 間違って解釈した場合、どんな損失が起きるのか。

### 6. 名前のアンカリングを疑う

既存名に引っ張られると、新しい責務や境界を発見できません。一度、現在のクラス名やテーブル名を隠し、ユースケースと目的から概念名を出し直します。

## 実行手順

1. 対象ユースケースのアクター、目的、文脈、ルールを記述する。
2. 仕様、会話、コードから重要語を抽出する。
3. 各語について、使用者、意味、例、反例、ルール、適用範囲を記録する。
4. 同じ語の意味をアクター・文脈ごとに比較する。
5. 意味が違う語を、目的が分かる別名へ分ける。
6. 新しい用語が示すモデル、イベント、操作、境界を特定する。
7. コード、API、DB、画面、テストの名前との差分を作る。
8. ドメインエキスパートと、具体例を使って認識を確認する。
9. 用語変更をIDE等の安全なRenameで反映し、文字列検索だけに頼らない。
10. 用語集を更新し、コンテキスト間の翻訳規則を残す。

## 完了条件

- [ ] 重要語にアクター、目的、文脈、ルールが対応している。
- [ ] 一般語の意味を、開発者の常識だけで確定していない。
- [ ] 同じ語の複数意味が可視化されている。
- [ ] 各meaningにstableなterm IDがあり、曖昧語と翻訳の参照先が解決している。
- [ ] 目的が異なる意味には、区別可能な名前または境界がある。
- [ ] 用語がコード、テスト、API、会話に一貫して反映される計画がある。
- [ ] 例と反例によって意味を検証している。
- [ ] コンテキスト間で異なる用語を、無理に一つへ統一していない。
- [ ] 名前変更が構造上の責務発見へつながっている。

## 失敗パターン

- 辞書的定義を唯一の正解として押し付ける。
- 用語集を作るだけで、コードやモデルを変更しない。
- 名前を長く説明的にするだけで、異なる責務を分割しない。
- 全社で同じ用語を一意に統一しようとする。
- エンジニアだけで定義し、実際の業務用法を確認しない。
- AIが提案したもっともらしい名前を、根拠なしに採用する。
- Renameを手作業の検索置換で行い、参照漏れを作る。

```text
- 一般的な意味と、この業務での意味を区別する。
- 同じ語が複数の意味や制約を持つ箇所を特定する。
- 各meaningにstableなterm IDを付け、曖昧語と翻訳からcanonicalなterm recordを参照する。
- 目的と役割が分かる用語を提案し、適用するコンテキストを示す。

```
