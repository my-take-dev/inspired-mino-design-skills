# 目的から抽象化を設計する

## このノウハウの核

抽象化は、似たコードをまとめたり、具体名を汎用名へ置き換えたりすること自体ではありません。複数の具体例から、**ある目的を達成するうえで本質的な役割・規則・契約を取り出し、目的に不要な詳細を捨てること**です。

目的駆動の抽象化では、二つの方向を区別します。

1. **問題領域の抽象化**: アクターの目的、目的達成を妨げる問題、制約、判断を整理し、業務上の概念を発見する。
2. **解決領域の抽象化**: 具体的な実装方法の違いを隠し、目的を満たす役割と契約をinterface、Policy、Strategyなどとして表現する。

抽象化の価値は「共通化した量」ではなく、目的を理解しやすくし、目的ごとの変更を局所化し、新しい具体例を安全に受け入れられるかで評価します。

## 解決する問題

- 見た目が似たコードを共通化した結果、片方だけの変更が難しくなる。
- `Common`、`Base`、`Abstract`、`GenericProcessor`など、目的が分からない抽象が増える。
- 業務概念が技術的なRepository、Handler、DTOだけで表現されている。
- 具体実装を知る呼び出し側が多く、種類追加で広範囲を編集する。
- 将来使うかもしれないという理由で、利用例のない拡張点を作る。
- 一つの抽象が複数のアクター・目的を含み、引数とフラグが増える。
- AIが重複コードを見つけるたびに、意味を確認せず共通化を提案する。

## 必要な入力

- アクター、目的、ユースケース、成功条件。
- 抽象化候補となる複数の具体例。
- 各具体例の入力、出力、副作用、失敗、制約。
- 変更履歴と、同時に変更された・別々に変更された事例。
- 既存のinterface、基底クラス、共通関数、汎用モジュール。
- 今後追加が見込まれる具体例と、その根拠。
- 呼び出し側が必要とする操作と、知らなくてよい詳細。

具体例が一つしかない場合は、抽象化を急がず、目的を明確にした具体実装を優先します。ただし外部契約、テスト境界、技術依存の隔離など、別の品質要件がある場合は抽象を検討できます。

## 判断規則

### 1. 抽象化の前に目的を一文で定義する

抽象の候補について、次の形式で目的を記述します。

```text
この抽象は、<利用者・呼び出し側> が
<具体的な実現方法> を知らずに、
<望む結果> を得られるようにする。
```

この文を具体的に書けない場合、抽象化の軸が未確定です。

### 2. 形の類似と意味の同一性を分ける

二つのコードが同じアルゴリズムや構造を持っていても、目的や変更理由が違うなら、原則として同じ業務抽象へまとめません。

共通化候補は、少なくとも次を確認します。

- 同じアクターまたは同じ利用側の目的を支えるか。
- 同じ契約と不変条件を守るか。
- 同じ理由で一緒に変更されるか。
- 一方だけ変化する可能性が低いか。
- 共通名が業務上の意味を持つか。

同じ目的でも実装が違う場合は、interfaceやPolicyで抽象化する価値があります。異なる目的で実装が偶然同じ場合は、重複を残すほうが安全なことがあります。

### 3. 具体例から差異と本質を抽出する

抽象化候補を表にします。

| 観点 | 具体例A | 具体例B | 契約に含めるか |
|---|---|---|---|
| 利用者の目的 |  |  |  |
| 必須入力 |  |  |  |
| 結果 |  |  |  |
| 失敗 |  |  |  |
| 副作用 |  |  |  |
| 不変条件 |  |  |  |
| 実装詳細 |  |  |  |

すべてを共通項へ押し込まず、利用側が必要とする本質だけを契約に残します。実装固有の設定やデータがinterfaceへ漏れるなら、抽象の軸が不適切な可能性があります。

### 4. 問題領域と解決領域を混同しない

- `DiscountEligibility`、`ShippingPriority`のような業務上の判断は問題領域の概念です。
- `HttpClient`、`Repository`、`Serializer`のような実現手段は解決領域の概念です。

技術抽象だけでシステムを構成すると、業務目的がコードから消えます。逆に、すべての技術詳細をドメインモデルへ持ち込むと、目的と手段が混在します。境界で接続します。

### 5. 抽象は利用側が所有する契約として設計する

interfaceは具体実装の都合ではなく、利用側が必要とする目的と品質を表します。

- 操作名は業務上の結果を表す。
- 入力は契約に必要な最小情報にする。
- 戻り値、失敗、副作用を明示する。
- 実装固有型や設定値を漏らさない。
- 実装間で成立しないメソッドを無理に共通化しない。

### 6. 抽象化の導入条件を持つ

次のうち複数が成立する場合に導入価値が高まります。

- 二つ以上の具体例があり、同じ目的と契約を持つ。
- 新しい具体例の追加が現実的に予測される。
- 具体実装の選択・変更を利用側から隔離したい。
- 外部技術の変更可能性が高い。
- テストで代替実装が必要で、契約が安定している。
- 目的上の概念に名前を与えることで理解が改善する。

### 7. 抽象化を戻せるようにする

抽象化が理解を悪化させた場合に、具体実装へ戻せるよう変更範囲を限定します。大規模な共通基盤へ一度に移さず、一つの利用側・一つの具体例から検証します。

## 実行手順

1. 抽象化候補の利用者、目的、品質目標を特定する。
2. 現在の具体例と、現実的な追加候補を収集する。
3. 各例の入力、結果、失敗、副作用、不変条件、変更理由を比較する。
4. 形だけの類似と、目的・契約の同一性を分ける。
5. 問題領域の概念と解決領域の手段を分類する。
6. 利用側が必要とする役割と最小契約を記述する。
7. 抽象化しない案を含め、理解コスト、変更容易性、試験性を比較する。
8. 採用する場合、利用側が所有するinterfaceまたはPolicyを作る。
9. 一つの具体実装と呼び出し側を移行し、契約テストを作る。
10. 実装固有の情報が抽象へ漏れていないか確認する。
11. 新しい具体例の追加シナリオと、片方だけ変更するシナリオで検証する。
12. 抽象化が目的を隠す場合は、名前、境界、または抽象化自体を見直す。

## 完了条件

- [ ] 抽象化の目的と利用者が一文で説明されている。
- [ ] 少なくとも具体例ごとの契約と変更理由を比較している。
- [ ] コード形状の類似だけで共通化していない。
- [ ] 問題領域と解決領域の抽象を区別している。
- [ ] interfaceまたはPolicyが利用側の目的を表している。
- [ ] 実装固有の型、設定、分岐が契約へ漏れていない。
- [ ] 抽象化しない案との比較がある。
- [ ] 新しい具体例追加と、片方だけの変更の両シナリオで有効性を確認している。
- [ ] 契約テストまたは同等の検証がある。
- [ ] 将来予測だけを根拠に過剰な拡張点を作っていない。

## 失敗パターン

- 重複を見つけたら即座に共通関数へする。
- `Base`クラスへ差分用フックとnullable fieldを増やし続ける。
- 一つのinterfaceに、実装によって意味のないメソッドを並べる。
- 技術語だけの抽象により、業務目的をコードから消す。
- 利用側ではなく具体実装側が契約を決め、内部都合を漏らす。
- 一つしかない具体例を、根拠なく多数の層で包む。
- 「再利用できるかもしれない」を目的として、実際の変更シナリオを示さない。
- 抽象を増やした結果、処理を追うためのファイル移動が増える。

