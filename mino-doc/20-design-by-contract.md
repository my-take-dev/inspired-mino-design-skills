# 契約による設計

## ノウハウの核

契約による設計は、自然言語の要件を、プログラムが守るべき検証可能な条件へ変換する方法です。

- **事前条件**: 処理を呼び出してよい入力・状態
- **事後条件**: 正常終了後に必ず成立する結果・状態
- **不変条件**: オブジェクトや集約が有効である限り常に成立する条件

ミノ駆動氏は、AIに高品質なテストを実装させる例として、対象メソッドの事前条件・事後条件・不変条件を検証し、Given-When-Thenで書くよう明示しています。重要なのはテスト記法ではなく、要件を契約へ変換し、AIが解釈で省略できない形にすることです。

## なぜAI開発の再現性に効くのか

生成AIは同じ要求でも異なる内部構造や命名を選び得ます。実装の形を固定するだけでは再現性は得られません。

一方、契約を固定すると、実装が異なっても次を機械的に判定できます。

- 無効入力を受理していないか
- 正常時に求める結果を返しているか
- 失敗時に状態を壊していないか
- どの公開操作を通っても不変条件を維持しているか

したがって、ここでいう再現性とは「毎回同じコード」ではなく、**毎回同じ契約を満たすコード**です。

## 契約抽出の入力

Skillは最低限、次を読みます。

- 機能要件と受入条件
- 業務ルール、計算規則、上限・下限
- 状態遷移と禁止遷移
- 既存API、DB制約、イベント契約
- エラー時、取消時、再実行時の振る舞い
- 時刻、通貨、数量、権限などの意味

情報が見つからない場合、勝手に一般的な値を補わず、`unknown` と仮定を分離します。

## 契約表

実装前に次を作ります。

| Operation | Preconditions | Postconditions | Invariants preserved | Failure behavior | Evidence |
|---|---|---|---|---|---|
| 注文確定 | 注文が未確定、明細が1件以上 | 確定状態になり合計金額が固定される | 数量・金額・通貨整合性 | 状態を変更せず理由を返す | 要件ID、仕様、既存テスト |

契約はメソッド単位だけでなく、次のレベルで作れます。

- Value Object
- Entity / Aggregate
- Application use case
- API endpoint
- Repository
- Message / event
- バッチ処理

## 実行手順

### 1. 要件を観測可能な文へ変換する

悪い例:

- 適切に注文を検証する
- エラーを考慮する

良い例:

- 数量が1未満または200超なら注文を生成しない
- 残高不足時は決済記録と注文状態を変更しない
- 同じ冪等キーの再要求は同じ結果を返し、二重決済しない

### 2. 条件を分類する

- 呼び出し側が満たすべきもの → 事前条件
- 処理が保証すべきもの → 事後条件
- モデル自身が常時守るもの → 不変条件
- 外部依存の可用性など → 環境条件

環境条件をドメイン不変条件へ混ぜません。

### 3. 契約の所有者を決める

契約を複数箇所へ重複実装しません。

- 値そのものの妥当性 → Value Object
- 集約内の整合性 → Aggregate
- ユースケース順序、権限、外部連携 → Application service
- 永続化形式 → Repository implementation / schema
- 通信形式 → API / message boundary

入力DTOのvalidationだけでドメイン不変条件を保証したことにしません。別経路からモデルを生成すると破られるためです。

### 4. 実装より先に契約テストを設計する

最低限、次を用意します。

- 事前条件の境界直前・境界値・境界直後
- 正常系の事後条件
- 失敗後に変更されない状態
- すべての公開操作後の不変条件
- 状態遷移の許可・禁止
- 冪等性、重複、順序逆転が重要ならその検証

### 5. 実装候補を契約に照合する

AIの説明ではなく、テスト結果とコード上の責務配置を確認します。

## Given-When-Thenの使い方

```text
Given: 契約の事前状態
When: 一つの公開操作
Then: 事後条件と維持された不変条件
```

一つのテストで多数の振る舞いを隠さず、契約項目とテストを追跡可能にします。

## 出力契約

```yaml
contract_analysis:
  subject: string
  operations:
    - operation: string
      preconditions: []
      postconditions: []
      invariants: []
      failure_behavior: []
      owner: string
      evidence: []
      tests: []
  unknowns: []
  contradictions: []
  implementation_gate:
    status: ready | blocked
    reasons: []
```

## 完了条件

- 各要件が少なくとも一つの契約項目へ追跡できる。
- 各契約項目に所有者が一つある。
- 正常時だけでなく失敗時の状態が定義されている。
- 不変条件がDTOやUIだけに依存していない。
- 境界値・禁止遷移・再実行を含むテストがある。
- テストが実装詳細ではなく観測可能な契約を検証する。

## 失敗パターン

- 既存コードの動きを無条件に契約とみなす
- 事前条件と入力サニタイズを混同する
- 例外が出れば失敗条件を満たしたことにする
- 正常系の戻り値だけテストし、状態破壊を見ない
- private methodの形にテストを結合する
- 同じ不変条件をController、Service、Entityへ重複記述する
- AIが列挙した契約を仕様確認なしで確定する

## Skill用プロンプト骨子

```text
対象要件とコードから契約表を作成してください。
各公開操作について事前条件、事後条件、不変条件、失敗時の状態、契約所有者、根拠を示してください。
要件に根拠のない条件は推測せず unknown としてください。
実装前にGiven-When-Then形式の契約テストを設計し、すべての要件が契約とテストへ追跡可能か監査してください。
```
