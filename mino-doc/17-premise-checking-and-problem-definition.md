# 前提を点検し、解くべき問題を定義する

## このノウハウの核

AI時代の開発では、設計や実装の速度より先に「何を問題として定義したか」の正確さが成果を左右します。誤った前提、曖昧な言葉、自分の経験だけに基づく解釈から問題を定義すると、AIはその誤りを高速かつ大規模に実装します。

Skillは、依頼をすぐ解決策へ変換せず、次を分離します。

- **観測事実**: 実際に見聞き、計測、記録できるもの。
- **解釈**: 観測事実へ与えた意味。
- **前提**: 解釈や判断が成立するため暗黙に置いている条件。
- **仮説**: 問題の原因や改善方法について検証可能な推測。
- **問題**: 特定のアクターが目的を達成できない状態。
- **解決策**: 問題を改善する候補の手段。

人間は問題定義のpilotとして、顧客要求、提供価値、前提、成功条件を決めます。AIは問題定義を補助し、問題解決では設計・実装案を生成します。人間は最終的に成果物の妥当性を検証します。

## 解決する問題

次の症状があるときに使います。

- 依頼を受けた直後に、技術や実装案が確定している。
- 「ユーザー」「類似」「適切」「高速」など、重要語の意味が人によって違う。
- 実装は仕様どおりなのに、利用者や事業側から「欲しかったものと違う」と言われる。
- 新しい項目を既存モデルへ追加するだけで済ませ、概念の違いを検討していない。
- AIが実装を繰り返すが、目的や価値が改善しない。
- 問題の原因と、現れた症状を混同している。
- 過去の経験や既存構造を「当然の前提」として疑っていない。

単純な不具合で再現条件、期待値、原因、修正範囲が既に確認されている場合は、前提監査を必要最小限にします。

## 必要な入力

- 依頼者の原文、会話記録、PRD、チケット。
- 対象アクター、利用場面、業務目的、提供価値。
- 実際の利用データ、障害、問い合わせ、観察結果。
- 現在のコード、モデル、テスト、API、schema、運用手順。
- 過去の意思決定と、その時点で置かれた制約。
- 既に提案されている解決策と、その根拠。
- 変更しない契約、期限、法令、セキュリティなどの制約。

## 認識の歪みを点検する観点

### 1. 言葉の意味は文脈で変わる

「類似商品」が、同じ種類の商品を指すのか、一緒に使う関連商品を指すのかは、アクターの目的と利用場面で変わります。

重要語ごとに次を確認します。

- 誰が使った言葉か。
- 何を達成したくて使ったか。
- どの状況で成立する意味か。
- どの条件を満たすものをその語に含めるか。
- 含めないものは何か。

### 2. 自分のスキーマで相手を解釈していないか

人は未知のものを、自分の知識や経験の枠組みへ当てはめて理解します。高速な判断に役立つ一方、相手の意図を自分の文脈で誤読する原因にもなります。

Skillは、少なくとも次の別解釈を作ります。

- 依頼者の立場からの解釈。
- 利用者の立場からの解釈。
- 運用・保守担当者の立場からの解釈。
- 現在のコードが暗黙に表現している解釈。

### 3. 既存モデルを現実そのものとみなしていないか

既存の`User`クラスに会社情報を足せるからといって、個人顧客と法人担当者が同じ概念とは限りません。既存構造は過去の判断結果であり、正しい前提の証拠ではありません。

### 4. 手段を問題として表現していないか

- 悪い問題定義: `マイクロサービス化できていない`
- 改善した問題定義: `注文ルールの変更で無関係な決済・配送機能まで同時変更が必要になり、リリース頻度が下がっている`

技術名を除いても、誰の何が困っているかを説明できる必要があります。

## 判断規則

### 1. 観測事実と評価語を分離する

```text
評価: このクラスは複雑すぎる
観測: 1つの変更で6種類のユースケースと3つの外部連携を同時修正する必要がある
```

「ひどい」「使いづらい」「遅い」などの評価は、その根拠となる観測へ変換します。

### 2. 前提には反証条件を付ける

```text
前提: 顧客は同じ種類の商品を比較したい
反証: 購入後のクリック・追加購入が関連商品へ集中している
```

反証不能な前提を、確定要件として扱いません。

### 3. 一つの解釈で確定しない

重要な要求には、最低でも次を作ります。

- 現在の第一解釈。
- 合理的な別解釈。
- どの証拠で両者を区別できるか。

ただし、可能性を無制限に列挙して作業を止めません。目的と結果を分岐させる解釈へ絞ります。

### 4. 症状から責務と不変条件まで遡る

問題を次の因果で整理します。

```text
観測された症状
  → 破られている目的・品質特性
  → 破られているルール・不変条件
  → 誤った責務所有者またはsource of truth
  → 構造上の原因
```

呼び出し側の例外追加など、症状を隠すだけの回避策を根本解決としません。

### 5. 問題定義と解決策選定を分ける

問題定義が合意される前に、特定技術を採用しません。解決策は複数候補とし、目的、品質特性、制約への適合で比較します。

### 6. 問い合わせは結果を分岐させるものに限定する

不足情報をすべて質問しません。次に該当するものを優先します。

- 対象アクターが変わる。
- 目的や成功条件が変わる。
- 公開契約またはデータ意味が変わる。
- セキュリティ、法令、金銭、安全へ影響する。
- 採用アーキテクチャが大きく変わる。

低リスクの不足は仮定と検証方法を明記して進めます。

## 実行手順

### Phase 1: Decision frame

1. 今回決めること、決めないこと、期限、決定者を記録する。
2. 可逆な判断と不可逆な判断を分ける。
3. 既に提示されている解決策を、問題ではなく候補手段として退避する。

### Phase 2: Evidence inventory

4. 依頼原文、利用データ、コード、テスト、問い合わせを収集する。
5. 各情報を`confirmed`、`inferred`、`assumption`、`unknown`、`contradiction`へ分類する。
6. 評価語を観測可能な事実へ書き換える。

### Phase 3: Meaning audit

7. 重要語と曖昧語を抽出する。
8. アクター、目的、文脈、ルールごとに意味を確認する。
9. 自分の解釈とは異なる合理的な別解釈を作る。
10. 解釈を区別するため必要な証拠または質問を定義する。

### Phase 4: Premise audit

11. 現在の判断が依存する前提を列挙する。
12. 各前提について根拠、反証条件、誤っていた場合の影響を記述する。
13. 既存コード、組織慣行、採用技術を正当性の根拠にしていないか確認する。
14. 利用者、運用者、事業、セキュリティの視座から反論する。

### Phase 5: Problem definition

15. 対象アクターの現在状態、望む状態、阻害要因を記述する。
16. 症状、破られたルール、責務所有者、構造原因を因果で接続する。
17. 技術名を使わず問題文を書き、解決すべき範囲と対象外を明示する。
18. 成功条件、失敗条件、維持契約を定義する。

### Phase 6: Solution handoff

19. 少なくとも二つの解決候補を作る。
20. 目的、品質特性、制約、移行、検証で比較する。
21. 問題定義、採用理由、未決事項を後続の設計・実装Skillへ渡す。
22. 実装後、元の問題と成功条件が改善されたかを検証する。

## 良い問題定義の形式

```text
[アクター] は [文脈] において [目的] を達成したい。
しかし [観測可能な阻害要因] により [具体的な損失・失敗] が生じている。
原因仮説は [破られたルール / 誤った責務 / 構造原因] である。
[成功条件] を満たし、[維持契約] を壊さず改善する必要がある。
```

問題文には、採用するライブラリ、パターン、クラス構成を原則として含めません。

## 完了条件

- [ ] 観測事実、解釈、前提、仮説、解決策が分離されている。
- [ ] 主要な用語が、アクター、目的、文脈、ルールに基づいて解釈されている。
- [ ] 結果を分岐させる別解釈が少なくとも一つ検討されている。
- [ ] 主要前提に根拠、反証条件、誤り時の影響がある。
- [ ] 問題文から特定技術を除いても、困りごとと価値を説明できる。
- [ ] 症状から破られたルール、責務、構造原因まで因果が示されている。
- [ ] 成功条件と維持契約がpass / fail判定できる。
- [ ] 採用解決策以外の合理的候補とトレードオフが記録されている。
- [ ] 実装後に元の問題が改善したか検証する方法がある。

## Hard gates

次の場合、問題解決フェーズへ進みません。

- 対象アクターまたは目的が特定できない。
- 重要語の解釈差で成果物が大きく変わるが、確認手段がない。
- 観測事実がなく、評価や推測だけで問題を断定している。
- 公開契約、データ意味、金銭、安全に関する前提が未確認。
- 問題文が特定技術の導入そのものになっている。
- 成功条件が「実装する」「リファクタリングする」で終わっている。

## 失敗パターン

- `なぜ`を無限に繰り返し、今回の意思決定に不要な抽象論へ広げる。
- すべてを疑って何も決めず、分析麻痺に陥る。
- 自分の経験と一致する解釈だけを採用する。
- 一般的なベストプラクティスを、対象システムの事実より優先する。
- 依頼者の発言をそのまま要件とし、背景目的を確認しない。
- コード上のモデル名を、業務上の正しい概念だとみなす。
- AIに問題定義を全面委任し、人間が価値判断を放棄する。
- 解決策の魅力に引かれ、問題定義を後から都合よく書き換える。

## エージェント向けプロンプト骨子

```text
この依頼をすぐ実装せず、前提監査と問題定義を行ってください。

1. 観測事実、解釈、前提、仮説、提示済み解決策を分ける。
2. アクター、目的、文脈、ルールから重要語の意味を確認する。
3. 第一解釈とは異なる合理的な別解釈と、それを区別する証拠を示す。
4. 主要前提へ根拠、反証条件、誤り時の影響を付ける。
5. 症状→破られた目的・品質→ルール・不変条件→責務所有者→構造原因の因果を示す。
6. 技術手段を含まない問題文、成功条件、維持契約、対象外を作る。
7. 複数の解決候補を比較し、問題定義がreadyになるまで実装しない。
```
