# データ破壊駆動で異常系と制約を発見する

## このノウハウの核

正常系からだけ設計すると、データがどのように壊れ、どのような損失へつながるかを見落としやすくなります。データ破壊駆動では、更新系ユースケースに対して意図的に「どうすれば不正な状態を作れるか」を考え、抜けている制約、更新順序、所有者、テストを発見します。

これは侵入手法の列挙を目的とせず、業務データ全般の異常系を設計へ取り込むための思考実験です。外部からの不正入力だけでなく、内部ロジック、並行更新、再試行、連携失敗によって壊れる場合も対象にします。

## 解決する問題

- 正常系を先に作り、障害発生後に場当たり的なバリデーションを足している。
- マイナス値、上限超過、重複、順序逆転などで不正状態を作れる。
- Controllerだけに入力検証があり、内部呼び出しやバッチが迂回する。
- 複数項目の組み合わせ、操作順序、並行処理の異常をテストしていない。
- 失敗時に部分更新、二重送信、外部副作用が残る。
- 「型があるから安全」「DB制約があるから安全」と思い込み、業務上の破壊を見ない。

## 対象と非対象

主に、作成、更新、削除、承認、確定、決済、割当など、状態を変える処理を対象にします。参照系も情報漏えい等の観点では重要ですが、このSkillの中心はデータ整合性と更新契約です。

本番環境の実データへ破壊操作を実行しません。思考実験、テスト、隔離された検証環境、dry-runを使います。

## 必要な入力

- 更新系ユースケースと状態遷移。
- 入力項目、型、範囲、相互関係。
- 事前条件、事後条件、不変条件。
- 全writer: API、画面、バッチ、管理ツール、イベント購読、移行処理。
- トランザクション境界、外部副作用、再試行方式。
- DB制約、メッセージ重複排除、ロック方式。
- 過去の不正データ、障害、補正作業。

## 破壊観点カタログ

### 値の破壊

- 負数、ゼロ、上限超過、最小値未満。
- オーバーフロー、桁落ち、丸め誤差。
- null、空文字、空集合、空白のみ。
- 形式上は正しいが業務上あり得ない値。
- 未知の列挙値、廃止済みコード。
- 日時の逆転、未来・過去の不正、タイムゾーン差。

### 関係の破壊

- 存在しない参照先。
- 所有者の違うデータの組み合わせ。
- 合計と明細、数量と在庫、通貨と金額の不一致。
- 同一性が必要な値の食い違い。
- 一意であるべきものの重複。

### 状態遷移の破壊

- 必須手順を飛ばす。
- 完了後に前状態へ戻す。
- 同じ操作を二度実行する。
- 古い状態から更新する。
- 取消後に確定する。
- 親より先に子を削除する。

### 時間・並行性の破壊

- 同時更新によるlost update。
- イベントの重複、遅延、順序逆転。
- タイムアウト後の再試行で二重実行。
- キャッシュと正データの不一致。
- 期限判定の境界時刻。

### 原子性の破壊

- DB更新だけ成功し、外部送信に失敗。
- 複数テーブルの一部だけ更新。
- 途中例外後にメモリまたは永続状態が残る。
- 補償処理がさらに失敗する。

## 判断規則

### 1. 破壊ケースを業務損失まで追跡する

「負数を受け付ける」で止めず、計算、保存、請求、在庫、通知へどう伝播し、誰にどんな損失が起きるかを記述します。

```text
-1個の注文
→ 合計金額が負になる
→ 返金処理として解釈される
→ 不正な値引きまたは経理処理が発生する
```

### 2. 検証を入口ごとに複製しない

破壊を防ぐ業務ルールは、すべてのwriterが通る権威あるモデルまたはサービスへ置きます。UIやAPIの入力検証は、利用者への早いフィードバックとして併用します。

### 3. 単項目だけでなく組み合わせと順序を見る

各フィールドが個別に妥当でも、組み合わせで不正になることがあります。状態遷移と複数操作のシーケンスをテストします。

### 4. 拒否後の状態も検証する

異常入力がエラーになっただけでは不十分です。

- 永続状態が変わっていない。
- 外部送信、課金、通知が行われていない。
- 再試行可能性が明確である。
- 監査ログやエラーが必要な範囲で残る。

### 5. 防御を多層化するが正のルールは一つにする

型、ドメインモデル、DB制約、冪等性キー、トランザクション、監視を組み合わせます。ただし、同じ業務判断を複数箇所へ独立実装せず、正の所有者を一つにします。

## 実行手順

1. 更新系ユースケースと全writerを列挙する。
2. 正常な事前状態、操作、期待する事後状態を記述する。
3. 値、関係、状態遷移、時間、並行性、原子性の観点で破壊ケースを出す。
4. 各ケースについて、生成経路と伝播先を追跡する。
5. 顧客、事業、会計、運用、セキュリティへの影響を評価する。
6. 防ぐべき事前条件、事後条件、不変条件を定義する。
7. 制約の権威ある所有者と、多層防御を決める。
8. Given-When-Thenの異常系テストと、失敗後不変のテストを作る。
9. 既存データに同種の破壊がないか読み取り専用で調査する。
10. 修正後、全writerが防御を迂回できないことを確認する。

## 完了条件

- [ ] 更新系ユースケースと全writerが対象になっている。
- [ ] 値だけでなく、関係、順序、時間、重複、並行性、部分失敗を検討している。
- [ ] 各破壊ケースが業務上の影響まで追跡されている。
- [ ] 防ぐべき契約と権威ある所有者が定義されている。
- [ ] エラー応答だけでなく、失敗後に状態と副作用が変わらないことを検証している。
- [ ] UIやControllerを迂回する経路にも防御がある。
- [ ] 既存データの違反を読み取り専用で確認している。
- [ ] 本番データへ破壊操作を実行していない。

## 失敗パターン

- 境界値テストだけを作り、操作順序や部分失敗を見ない。
- セキュリティ診断と同一視し、内部ロジック由来の不整合を無視する。
- 各入口へ同じif文を追加する。
- エラーを返せば完了とし、保存や外部送信済みか確認しない。
- 破壊ケースを無制限に列挙し、影響と優先度を付けない。
- 本番環境で試す。
- 制約追加による既存データ・互換性の問題を見落とす。

## エージェント向けプロンプト骨子

```text
対象の更新系ユースケースに対し、データを壊す思考実験を行ってください。

- 全writerと正常な状態遷移を確認する。
- 値、項目間関係、状態遷移、時間、重複、順序逆転、並行更新、部分失敗の破壊ケースを列挙する。
- 各ケースを、入力または操作→内部伝播→永続化・外部副作用→業務損失の順で追跡する。
- 必要な事前条件、事後条件、不変条件と権威ある所有者を提案する。
- 多層防御を示すが、業務ルールの正の実装は一つにする。
- 拒否後に状態と副作用が変わらない異常系テストを作る。
- 実データへの破壊操作はせず、既存違反は読み取り専用で調査する。
```
