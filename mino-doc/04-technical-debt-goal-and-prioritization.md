# 技術的負債のゴールと優先順位を定義する

## このノウハウの核

技術的負債は、単に古いコード、長いクラス、警告件数ではなく、**目的に適したあるべき構造と現在の構造の差**として捉えます。あるべき姿が定義されていなければ、何が負債か、どこまで直せば完了かを判定できません。

負債解消は全面清掃ではありません。限られた資源の中で、事業上重要で、今後よく変わり、現在の構造が変更を妨げている領域を優先します。

## 解決する問題

- 静的解析の警告件数を、そのまま負債総量として扱っている。
- 目につく巨大クラスから無計画に着手する。
- リファクタリングの完了条件がない。
- 変更されないコードを美しくする一方、重要なホットスポットが放置される。
- 負債解消と設計スキル育成が別々に進み、新しい負債が増え続ける。
- 経営、プロダクト、開発の間で優先順位を説明できない。

## 必要な入力

### 事業・プロダクト側

- 中長期ロードマップと予定される変更。
- コアドメイン、主要顧客価値、障害時の事業影響。
- 現場の業務担当者、ドメインエキスパートへの聞き取り。

### 技術側

- 変更頻度と変更失敗の履歴。
- 障害、手戻り、修正漏れ、調査時間の事例。
- 巨大クラス、分岐、重複、循環依存などの静的兆候。
- git の変更ホットスポット、共同変更ファイル、所有者不明領域。
- テスト不足、データ不整合、レイヤ混在、責務混在。
- エンジニアが「触りたくない」と感じる箇所の聞き取り。

静的指標、履歴、現場の痛み、事業重要度を組み合わせ、一つの指標だけで判定しません。

## 判断規則

### 1. 先にあるべき姿を定義する

対象領域ごとに次を定義します。

- アクターと目的。
- 守るべき仕様・制約・不変条件。
- 責務を所有するモデルまたはモジュール。
- 許容する依存方向。
- 変更が完結すべき境界。
- 必要なテストと観測方法。

「行数を半分にする」ではなく、「法人注文の変更が個人注文へ影響しない」のような状態をゴールにします。

### 2. 負債を因果連鎖で記述する

```text
観測された症状
→ 破られている品質目標または不変条件
→ 誤った責務所有者・境界
→ 構造的原因
→ 事業・開発への影響
```

例:

```text
割引変更のたびに配送処理が壊れる
→ 注文ルールの変更が目的境界内で完結しない
→ 注文と配送のロジックを巨大 Service が所有
→ 異なる目的の責務混在
→ リリース遅延と回帰障害
```

### 3. 優先順位は価値・変化・負債・リスクで決める

優先度の概念式:

```text
priority = business_criticality
         × expected_change
         × debt_impact
         × failure_risk
         ÷ remediation_cost
```

これは精密な財務計算ではなく、比較のための判断枠です。根拠のない点数を作るより、High / Medium / Low と説明を残します。

各候補では5因子を一度ずつ評価し、特に`debt_impact`は、現在の構造がproduct、delivery、operationへ生む影響、判断理由、Evidenceを分けて残します。最終的な優先順位は、product valueとtrade-offを判断できる人間のownerが決めます。

特に優先しやすいのは、次の組み合わせです。

- コアドメインである。
- 近い将来、変更が多い。
- 現在の変更失敗、調査工数、障害が大きい。
- 段階的に改善でき、早期に効果を確認できる。

### 4. 負債指標を「兆候」と「証拠」に分ける

- 兆候: 行数、複雑度、重複、引数数、低いテスト率。
- 設計上の証拠: 異なる目的の混在、不変条件の分散、誤ったレイヤ、意図しない公開、変更境界の崩壊。
- 影響の証拠: 障害、変更失敗、長いリードタイム、複数チーム調整、修正漏れ。

兆候だけでは、直すべき理由と改善後の姿を確定しません。

### 5. 解消と再発防止を対にする

負債を直すだけでなく、次を同時に計画します。

- 設計ガイドラインまたはレビュー観点。
- Characterization test と回帰防止。
- 実コードを使った設計学習。
- 負債の定期計測とトレンド確認。
- 新規変更が同じ責務混在を作らない受け入れ条件。

## 実行手順

1. 対象領域と期間を限定する。
2. コアドメイン、ロードマップ、変更頻度を調べる。
3. 静的指標、git 履歴、障害、現場ヒアリングから候補を収集する。
4. 各候補の目的、あるべき境界、品質目標を定義する。
5. 現在との差を因果連鎖で記述する。
6. 価値、変化、負債影響、失敗リスク、解消コストを評価する。
7. 上位候補について、最小の安全な改善単位と検証方法を設計する。
8. 先行指標と結果指標を定義する。
9. 解消後の再発防止策を組み込む。
10. 定期的に再評価し、ロードマップ変化に合わせて順序を更新する。

## 完了条件

- [ ] 負債解消のあるべき姿が、目的と品質特性から定義されている。
- [ ] 各負債項目に、コード上の根拠と実際の影響がある。
- [ ] 優先順位が、事業重要度、今後の変化、負債影響、失敗リスク、解消コストと、そのEvidenceから説明されている。
- [ ] 最終順位を判断する人間のownerが明示され、不明なら確認方法が残っている。
- [ ] 静的指標だけで重大度を決めていない。
- [ ] `do-not-fix` または後回しの判断も明示されている。
- [ ] 上位項目に安全な段階、テスト、停止条件がある。
- [ ] 再発防止策が負債解消と対になっている。
- [ ] 完了条件が「コードを整理した」ではなく観測可能な状態である。

## 失敗パターン

- 巨大なファイルを見つけた順に着手する。
- コアでも変化もない領域を、見栄えのために全面改修する。
- 負債スコアを絶対的な真実として扱う。
- リファクタリングの効果を行数削減だけで示す。
- 全面刷新を唯一の解決策にする。
- 新機能の納期と競合するのに、段階導入や共同計画を作らない。
- 個人の設計能力不足だけを原因にし、境界、レビュー、教育、ツールの問題を無視する。

## エージェント向けプロンプト骨子

```text
対象コードの技術的負債を、あるべき姿と現在の差として分析してください。

1. アクター、目的、品質目標、望ましい責務境界を定義する。
2. 静的兆候、変更履歴、障害、テスト、現場の痛みから負債候補を集める。
3. 各候補を「症状→違反された目標→誤った責務所有者→構造的原因→影響」の順で説明する。
4. 事業重要度、今後の変更頻度、負債影響、失敗リスク、解消コストを一度ずつ評価し、根拠と影響説明を残す。
5. 今直す、次に直す、後回し、直さないを区別する。
6. 上位項目には、挙動維持テスト、段階的変更、検証、再発防止を示す。
7. 根拠のないスコアや目的を作らず、最終順位は人間のownerへ委ねる。
```
