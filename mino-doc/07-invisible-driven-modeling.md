# 見えないものを起点にモデリングする

## このノウハウの核

人、商品、設備など、物理的に見える対象をそのまま一つのモデルにすると、対象が持ち得るあらゆる属性とロジックが集まり、巨大化しやすくなります。モデルは物理的な実体の写しではなく、**特定の目的を達成するための小さなシステム**として設計します。

モデリングで優先して発見するのは、目に見えない次の要素です。

- アクターの目的。
- アクターが置かれた状況。
- 目的達成を妨げる問題。
- 守るべき制約と方針。
- 判断、約束、権利、責任。
- 業務上の出来事と状態変化。

物理的対象は入力の一つであり、モデル境界を決める唯一の軸ではありません。

## 解決する問題

- `Product`、`User`、`Account` など、サービスの大通りにあるモデルが巨大化する。
- 状況ごとに不要な属性が大量に存在する。
- 目的の違うルールが同じモデル内で衝突し、フラグと分岐が増える。
- ER図や既存DBから始めたため、業務上の判断や制約がモデルに現れない。
- 現実世界の名詞を列挙しただけで「ドメインモデル」としている。
- 何を基準に巨大モデルを分けるべきか分からない。

## 必要な入力

- アクターと、各アクターが達成したい目的。
- ユースケース、業務プロセス、イベント、判断。
- 目的達成上の問題、例外、制約。
- 状況ごとに必要なデータと不要なデータ。
- 現在の巨大モデル、状態フラグ、条件分岐、利用箇所。
- ドメインエキスパートの語彙、具体例、失敗例。

## 判断規則

### 1. 物理名詞を見つけたら「どの目的で見るか」を問う

同じ商品でも、目的によってモデルは変わります。

```text
審査する目的 → 審査品目
注文する目的 → 注文品
在庫を維持する目的 → 在庫品
配送する目的 → 配送品
```

目的別モデルは、目的外の属性とルールを持ちません。

### 2. 目的から問題と必要情報を導く

モデル候補を作る順序:

```text
アクター
→ 目的
→ 目的達成を妨げる問題
→ 必要な判断・制約
→ 判断に必要な情報
→ 情報と振る舞いを持つモデル
```

既存テーブルの列を先に全部モデルへ写しません。

### 3. 状況の違いをモデルの違いとして検討する

次の状態で必要情報やルールが根本的に違う場合、単一モデルの状態フラグではなく別の概念・役割・ライフサイクルを検討します。

- 申請前と承認後。
- 個人顧客と法人顧客。
- 販売前と販売後。
- 見積もりと契約。
- 予約注文と即時注文。

一時的な状態遷移として同一の識別を守る必要がある場合は、Entity内の状態として扱う選択肢もあります。目的、ルール、ライフサイクルの独立性で判断します。

### 4. 「モデルに何を入れるか」より「何を入れないか」を決める

各モデルに `out_of_scope` を定義します。目的外の属性や操作を拒否できることが、モデルの単純さと変更容易性を守ります。

### 5. 見えない概念を明示的な型・イベント・ポリシーにする

次のような語が会話に現れたら、単なるコメントや条件式ではなくモデル候補として扱います。

- 資格、承認、予約、割当、上限、猶予、優先度。
- 見積、合意、請求、取消、失効。
- 判定、方針、戦略、責任、権限。

### 6. 可視化は議論の道具として使う

ユースケース図、イベントストーミング、ドメインモデル図を使い、目的が異なるアクターやルールが混ざっていないかを共同で確認します。図の美しさではなく、認識のずれを発見できることを重視します。

## 実行手順

1. 対象の巨大モデルまたは業務領域を選ぶ。
2. そのモデルを利用するアクターとユースケースを列挙する。
3. 各ユースケースについて、目的、問題、判断、制約、必要情報を記述する。
4. 同じ物理名詞に対して異なる役割名を仮置きする。
5. 役割ごとに、必要属性、振る舞い、不変条件、ライフサイクル、対象外を定義する。
6. 暗黙の概念を、Value Object、Entity、Policy、Eventなどの候補へ変換する。
7. 現在の巨大モデルの属性・メソッドを、目的別候補へマッピングする。
8. どの候補にも属さない要素は、別目的、技術関心事、不要コードの可能性を調べる。
9. ドメインエキスパートと具体例を使って検証する。
10. 代表的な変更要求で、目的外の要素を読まずに変更できるか確認する。

```yaml
  name: "商品"

  - actor: "購入者"
    purpose: "購入対象と数量を決める"
    context: "注文"

  - name: "注文品"
    purpose: ""
    problems_solved: []
    required_information: []
    behaviors: []
    invariants: []

  - current_symbol: "Product.price"
    target_concept: "注文品"
    rationale: ""

  - symbol: ""
    hypothesis: "別目的 | 技術関心 | 不要"

```

## 完了条件

- [ ] 物理名詞だけでモデル境界を決めていない。
- [ ] 各モデル候補にアクター、目的、解決する問題がある。
- [ ] 必要情報が目的と判断から導かれている。
- [ ] 各モデルに対象外が定義されている。
- [ ] 状況による意味・ルールの違いが明示されている。
- [ ] 暗黙の判断や制約が、名前のある概念として表現されている。
- [ ] 巨大モデルの全要素が、目的別候補、別関心、不要のいずれかへ分類されている。
- [ ] 代表的な変更が、一つの目的モデル内で完結する。

## 失敗パターン

- 現実世界に存在する名詞を、すべてクラスにする。
- 一つの物理対象には一つのモデルしか存在できないと考える。
- 目的別モデルへ分けた後、共通基底クラスへ全属性を戻す。
- 画面ごとのDTO分割だけで、業務ルールの所有者を変えない。
- 状態が一つ増えるたびに別モデルを作り、識別や一貫性を失う。
- 目的を推測だけで作り、業務担当者と確認しない。
- 図を完成させることが目的になり、コードとテストへ接続しない。

## エージェント向けプロンプト骨子

```text
対象領域を、物理的な名詞ではなく、目的・状況・問題・制約を起点にモデリングしてください。

1. 現在の中心モデルを利用するアクターとユースケースを列挙する。
2. 各ユースケースの目的、問題、判断、必要情報、制約を整理する。
3. 同じ物理対象を、目的別の役割モデルへ分ける候補を示す。
4. 各候補に必要属性、振る舞い、不変条件、ライフサイクル、対象外を定義する。
5. 会話や条件式に隠れた見えない概念を、名前のある型・ポリシー・イベントとして提案する。
6. 現在の属性・メソッドを候補へマッピングし、属さない要素を調査する。
7. 推測と確認済みの業務事実を区別する。
```
