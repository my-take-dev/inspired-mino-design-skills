# 目的・目標・手段を分離する

## このノウハウの核

ソフトウェア、機能、モジュール、クラス、メソッドは、それ自体が目的ではなく、誰かの望む状態を実現するための手段です。設計判断を始める前に、次を分離します。

- **目的**: アクターが実現したい状態、得たい価値、解消したい困りごと。
- **目標**: 目的を達成したと判定するための、観測可能な仕様、制約、成功条件。
- **手段**: 目標を満たすためのシステム、機能、モデル、コード、技術。

Skillは、依頼された手段をそのまま実装せず、上位目的との整合を確認してから設計へ進みます。

直近の公開資料で強調される「技術の引力」を避けるため、具体的な技術から目的へ遡るだけで終わらず、目的から観測可能な目標、契約、実装、テストへ戻る往復を必須にします。

## 解決する問題

次の症状があるときに使います。

- 「このライブラリを使う」「このクラスへ追加する」だけが決まり、理由が不明。
- 同じ言葉に複数の意味があり、要件やコードの解釈が揺れる。
- 技術選定が流行、慣例、既存実装の模倣だけで決まっている。
- 完成条件が「実装した」「リファクタリングした」になっている。
- AIへ具体的な命令をしたのに、期待と異なる成果物が返る。
- 抽象的な「顧客価値」「利益」まで遡ったが、具体的な設計判断へ戻れない。

単純な機械的変更で、目的、成功条件、影響範囲が既に明白な場合は、分析を過剰に広げません。

## 必要な入力

- 依頼文、PRD、ユースケース、受け入れ条件。
- 利用者、運用者、事業担当者などのアクター。
- 現在のコード、テスト、画面、API、データ、運用フロー。
- なぜ今変更するのかを示す事業上または運用上の背景。
- 維持すべき挙動と変更してよい範囲。
- 既に提示されている技術案と、その根拠。

不足している情報は、確認済み事実、妥当な仮説、未決事項に分けます。

## 判断規則

### 1. 目的は動詞を含む望ましい状態として書く

「決済」「User」「マイクロサービス」は目的ではありません。例:

- 悪い例: `ポイント機能を作る`
- 改善例: `継続利用の動機を作り、顧客が自社サービス群を再利用しやすくする`

目的の記述には、原則として「誰が」「どうなりたいか」を含めます。

### 2. 目標は真偽判定できる形にする

目的だけでは実装の正否を判定できません。目的から、機能要件、制約、事前条件、事後条件、不変条件へ落とします。

- 目的: 正しい数量で注文したい。
- 目標: 注文数量は1以上、販売上限以下である。
- 手段: 制約を生成時に保証する`OrderQuantity`型とテスト。

### 3. 手段を複数案として扱う

依頼に書かれた技術や構造は、目的を満たす候補の一つです。既存制約により固定されていない限り、少なくとも次を比較します。

- 現行構造を拡張する案。
- 責務を分離する案。
- 運用または仕様を変え、実装量を減らす案。

### 4. 上位目的と下位目的を接続する

目的は階層を持ちます。下位機能だけを最適化すると、主目的に寄与しない作り込みが発生します。

```text
事業目的
└── プロダクト価値
    └── 利用者の主目的
        ├── ユースケースの目的
        │   └── モジュール・実装の目的
        └── 別ユースケースの目的
```

各手段について「これが成功すると、どの上位目的が前進するか」を説明できるようにします。

### 5. `つまり`と`たとえば`で具体と抽象を往復する

具体的な技術から上位目的へ遡るときは、`つまり、何のためか`を一段ずつ問います。

```text
注文テーブルへ法人区分を追加する
→ つまり、法人注文を識別する
→ つまり、法人固有の承認・価格ルールを適用する
→ つまり、法人購買担当者が組織制約の中で安全に購入できる
```

目的から具体へ戻るときは、`たとえば、何がtrueなら達成か`を問います。

```text
法人購買担当者が安全に購入できる
→ たとえば、未承認の注文は確定できない
→ たとえば、承認金額を超える変更は再承認になる
→ たとえば、禁止遷移を契約テストで検証する
```

上へ遡るだけ、下へ具体化するだけの片道分析にしません。詳細な手順は`26-technical-gravity-and-abstraction-navigation.md`を参照します。

### 6. 抽象化の高さを意思決定に必要な範囲へ限定する

次のいずれかへ到達したら、無制限に`なぜ`を繰り返しません。

- 採用案と不採用案を比較できる目的が分かった。
- 成功条件と対象外を定義できる。
- それ以上遡っても今回の実装判断が変わらない。
- 権限外の事業判断へ入る。
- 根拠がなく、仮説だけになる。

### 7. 技術語を除いても問題と目的を説明できるようにする

`Redisを導入したい`、`マイクロサービス化したい`のような依頼は、候補手段として保持します。技術語を除いたとき、誰がどの文脈で何に困り、どの状態を望むかを説明できなければ、問題定義へ戻ります。

## 実行手順

1. 依頼に登場するアクターを列挙する。
2. 各アクターについて、要求された手段ではなく、望む結果を記述する。
3. 提示済みの技術案を`candidate_means`へ退避し、未選択として扱う。
4. `つまり`を使い、具体的な実装から少なくとも一段上の目的を確認する。
5. 各目的の根拠を`confirmed`、`inferred`、`assumption`、`unknown`へ分類する。
6. 目的達成を判定できる機能要件、制約、失敗条件を目標として列挙する。
7. `たとえば`を使い、目的から契約、品質シナリオ、実装候補、テストへ戻る。
8. 現在のコードや提案された技術を手段として対応付ける。
9. 目的が異なるのに一つの手段へ混在している箇所を検出する。
10. 目標を満たさない手段、主目的へ寄与しない手段、過剰な手段を指摘する。
11. 採用案と不採用案を、目的、品質特性、制約への適合で比較する。
12. 実装項目とテストを、対応する目的・目標へ追跡可能にする。
13. 抽象化の停止理由を記録する。

## 出力契約

Skillは最低限、次を出力します。

```yaml
actors:
  - name: "アクター名"
    context: "置かれている状況"
purposes:
  - id: P1
    actor: "アクター名"
    desired_state: "実現したい状態"
    parent_purpose: null
    evidence_status: confirmed | inferred | assumption | unknown
goals:
  - id: G1
    purpose: P1
    observable_condition: "pass/fail判定可能な条件"
    constraints: []
means:
  - id: M1
    candidate: "コード、モデル、技術、運用"
    status: unselected | selected | rejected
    supports: [G1]
    risks: []
abstraction_ladders:
  - concrete_item: "具体的な技術・実装"
    levels:
      - level: implementation
        statement: ""
      - level: capability
        statement: ""
      - level: actor_purpose
        statement: ""
      - level: product_value
        statement: ""
    stop_reason: decision_useful | authority_boundary | insufficient_evidence
misalignments:
  - evidence: "仕様またはコード上の根拠"
    issue: "目的と手段の不整合"
    proposed_boundary: "改善方向"
traceability:
  - purpose: P1
    goal: G1
    means: [M1]
    tests: []
open_questions: []
```

実装を伴う場合は、各変更とテストに`purpose_id`と`goal_id`を付けます。

## 完了条件

- [ ] 主要アクターが特定されている。
- [ ] 各目的が、特定のアクターと望ましい状態を示している。
- [ ] 各目的に少なくとも一つの観測可能な目標がある。
- [ ] 採用する手段が、どの目標を満たすか説明されている。
- [ ] 技術語を除いても、問題と目的を説明できる。
- [ ] 具体的な実装から目的へ遡り、目的から契約・テストへ戻っている。
- [ ] 上位目的の根拠状態が示されている。
- [ ] 手段自体を目的とする記述が除去または明示的に正当化されている。
- [ ] 異なる目的が一つのモジュールに混ざる場合、その理由と影響が示されている。
- [ ] 目的、目標、手段、テストが追跡可能である。
- [ ] 抽象化の停止理由がある。
- [ ] 未確認の目的を事実として断定していない。

## 失敗パターン

- `なぜ`を無制限に繰り返し、今回の意思決定に不要な事業戦略まで広げる。
- 「利益向上」だけを最上位目的とし、具体的な利用者価値を失う。
- 依頼者が指定した手段を無視し、根拠なく全面再設計する。
- 目的を抽象語で書き、成功条件を定義しない。
- コードから推測した目的を、ドメインエキスパートの確認なしに確定する。
- 抽象的な理念を作って満足し、契約、実装、テストへ戻らない。
- 技術を避けること自体を目的にし、必要な技術判断を先送りする。

## エージェント向けプロンプト骨子

```text
対象の仕様とコードを、目的・目標・手段に分けて分析してください。

1. アクターと文脈を特定する。
2. 提示済みの技術案を候補手段として退避する。
3. 各具体要素から「つまり何のためか」で、実装目的、機能目的、利用者目的、プロダクト価値へ遡る。
4. 上位目的に根拠状態を付け、推測を確定しない。
5. 「たとえば何がtrueなら達成か」で、目標、制約、契約、品質シナリオ、実装候補、テストへ戻る。
6. 目的と手段の不整合、異なる目的の混在、目的化した技術を根拠付きで指摘する。
7. 複数の改善案を目的と品質特性への効果で比較する。
8. 目的、目標、実装、テストのtraceabilityと、抽象化の停止理由を示す。
```
