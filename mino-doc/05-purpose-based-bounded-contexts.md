# 目的の違いから境界付けられたコンテキストを発見する

## このノウハウの核

同じ物理的な対象や同じ名詞を扱っていても、アクターと目的が違えば、必要なデータ、ルール、ライフサイクル、判断は異なります。一つの普遍的なモデルへ統合せず、目的が一貫する範囲を境界付けられたコンテキストとして分けます。

例として「商品」は、注文する人には注文品、配送する人には配送品、在庫を管理する人には在庫品です。これらを一つの巨大な商品モデルで表すと、異なる目的の制約が衝突します。

境界は、テーブル、画面、部署、既存サービスの都合だけで決めません。主に次の違いから発見します。

- アクター。
- 達成したい目的。
- 言葉の意味。
- 守るべきルールと不変条件。
- 業務プロセスと変化の周期。
- 意思決定の責任者。

## 解決する問題

- `User`、`Product`、`Order` などの中心モデルが巨大化している。
- 状態フラグによって、使える属性や処理が大きく変わる。
- 同じ言葉を部門ごとに異なる意味で使う。
- 一つの機能変更が、無関係なユースケースへ波及する。
- 「共通だから一つにする」という判断で、多目的モデルが作られる。
- マイクロサービス境界を既存組織図やテーブル単位で決めている。

## 必要な入力

- アクター一覧と各アクターの目的。
- ユースケース、業務イベント、コマンド、判断。
- 用語集と、用語を使う会話・仕様・画面。
- 事前条件、事後条件、不変条件。
- データの生成、更新、参照、破棄のライフサイクル。
- 現在のモジュール、DB、API、チームの依存関係。
- 仕様変更の頻度と、同時に変更される要素。

イベントストーミング、ユースケース分析、ドメインエキスパートへの聞き取りを入力収集に使えます。

## 判断規則

### 1. アクターと目的の組を起点にする

同じアクター名でも、状況によって役割が違う場合があります。単に「ユーザー」とせず、次の形で記述します。

```text
アクター: 倉庫担当者
文脈: 出荷準備中
目的: 正しい品目と数量を、指定された宛先へ出荷可能にする
```

目的が異なる組は、別コンテキスト候補です。

### 2. 意味またはルールが変わる場所を境界候補にする

次のどれかが変わるなら、同じ名詞を共有すべきか疑います。

- 識別子の意味。
- 必須属性。
- 有効な状態遷移。
- 計算、判断、制約。
- ライフサイクル。
- 正とするデータの所有者。

意味は同じだが表現だけが違う場合は、変換境界で済む可能性があります。意味そのものが違う場合は、モデルを分けます。

### 3. 一つの目的に一つの主要モデルを対応させる

一つのコンテキスト内では、モデル、用語、ルールが一貫します。別コンテキストの都合で、目的外の属性や分岐を追加しません。

ただし、コンテキストをクラス一個の細かさまで分割するのではなく、同じ言語とモデルが矛盾なく使える業務範囲として定義します。

### 4. 共通データと共通モデルを区別する

同じ商品IDを参照していても、注文品と配送品が同じモデルである必要はありません。共有する識別子や最低限の契約と、各コンテキストが所有する意味・ルールを分けます。

- 共通化してよい: 安定した識別、明確な交換契約。
- 共通化に注意: 可変な業務ルール、目的固有の状態、計算。

### 5. コンテキスト間関係を明示する

境界を作るだけでなく、次を決めます。

- 上流と下流。
- 共有する事実と所有者。
- 同期API、非同期イベント、バッチなどの連携方法。
- 翻訳または腐敗防止層の必要性。
- 整合性のタイミング。
- 失敗時の責任と再試行。

失敗時の責任だけでなく、再試行、重複、結果が確定しない場合を関係ごとに判定します。非該当には理由を、未確認には確認方法と未解決時の影響を残し、連携方式だけから業務上の失敗契約を推測しません。

### 6. 組織・デプロイ境界は後から選ぶ

境界付けられたコンテキストは意味とモデルの境界です。必ず別サービスにするという意味ではありません。まずモジュールとして分け、必要な独立性、スケール、運用、チーム所有に応じて物理分割を判断します。

## 実行手順

1. 主要な業務フローとイベントを並べる。
2. 各イベントについて、誰が、何のために、何を判断・変更するかを記述する。
3. 登場する用語の意味、必須属性、ルール、ライフサイクルをアクター別に比較する。
4. 目的または意味が変わる地点に境界候補を置く。
5. 各候補内で、用語と不変条件が一貫するか確認する。
6. 多目的な中心モデルを、目的別モデルへ仮分割する。
7. コンテキスト間で交換する最小の事実と所有者を決める。
8. 現在のコード、DB、API、チーム境界との差を分析する。
9. Context Map と依存方向を作る。
10. 代表的な変更シナリオで、変更が一つの境界内に収まるか検証する。

Context Map は Mermaid などで併記して構いませんが、図だけでなく、各境界の目的、言語、所有ルールを文章で残します。

## 完了条件

- [ ] 各コンテキストにアクターと目的がある。
- [ ] 同じ用語の意味が境界内で一貫している。
- [ ] 目的固有のルールとデータ所有者が明確である。
- [ ] コンテキスト間で交換する事実が最小化されている。
- [ ] 各関係に整合性timing、failure owner、retry、duplicate、ambiguous outcomeの適用判定があり、unknownの確認方法と影響が残っている。
- [ ] 多目的な共通モデルを、単なるDRYのために共有していない。
- [ ] 境界とサービス分割を同一視していない。
- [ ] 代表的な仕様変更が期待する境界内で完結することを説明できる。
- [ ] 境界候補が、既存組織図だけで決まっていない。

## 失敗パターン

- 名詞ごとにコンテキストを作る。
- CRUD画面やテーブル一つを一コンテキストとする。
- すべてのデータを複製し、所有者と同期規則を決めない。
- 境界を細かくしすぎ、単一の不変条件を複数境界へ分断する。
- 「共通」という名前のコンテキストへ業務ルールを集める。
- 現在のサービス構成を正解として、目的分析を省略する。
- 境界発見と同時に全面マイクロサービス化を計画する。

## エージェント向けプロンプト骨子

```text
仕様、業務フロー、コードから、目的に基づく境界付けられたコンテキスト候補を発見してください。

1. アクター、文脈、目的、業務イベントを整理する。
2. 同じ用語について、意味、必須属性、ルール、ライフサイクル、所有者を比較する。
3. 目的または意味が変わる地点を境界候補にする。
4. 各候補に、目的、ユビキタス言語、所有ルール、所有データ、対象外を定義する。
5. 多目的な中心モデルを目的別モデルへ分ける案を示す。
6. コンテキスト間で交換する最小の事実、上流下流、連携方式、翻訳、整合性timing、失敗時の責任を定義する。
7. 各関係のretry、duplicate、ambiguous outcomeを判定し、非該当理由またはunknownの確認方法を残す。
8. 現在のコード境界との差を根拠付きで示す。
9. サービス分割は別の意思決定として扱う。
```
