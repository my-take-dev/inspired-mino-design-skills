## Reservation confirmation and partial failure

```yaml
mode: design
required_platforms: []
raw_request: >
  $mino-reproducible-development を使い、予約確定ユースケースのImplementation Specを作ってください。
  codeは変更しません。
confirmed_evidence:
  - actorは予約を購入する顧客と、失敗取引を照合する運用担当者
  - quantityは1以上8以下、moneyはamountとJPY currencyを失わない
  - stateはpending、confirmed、consumed、cancelled、expired
  - pendingからconfirmed/cancelled/expiredへ遷移できる
  - confirmedからconsumed/cancelledへ遷移でき、consumed後のcancelは禁止
  - 同じidempotency keyと同じrequestのduplicate confirmは同じ結果を返し、二重authorizationしない
  - 同じkeyでrequest内容が違う場合はconflictにする
  - payment authorization成功後にDB saveが失敗する可能性がある
  - writerはREST API、admin batch、migration importer
  - readerはcheckout API、invoice batch、support console
  - payment providerは一つで、追加予定のEvidenceはない
  - provider SDK型を公開contractへ出してはいけない
known_unknowns:
  - authorization成功後にsave失敗した場合、void、refund、reconciliationのどれを採用するか
  - recoveryの業務ownerと許容時間
allowed_assumptions:
  - artifact内の安定IDと説明用の名前
prohibited_changes:
  - recovery policyを一般知識から確定する
  - provider追加用Strategyまたはfactoryを作る
  - test planを実行済みとする
  - repositoryを変更する
```

## Cross-context debt and delivery failure semantics

```yaml
mode: design
required_platforms: []
raw_request: >
  $mino-reproducible-development を使い、注文確定から配送受付までの既存連携について、
  技術的負債の優先度、context別の用語、context間の失敗契約を含むImplementation Specを作ってください。
  codeは変更しません。
confirmed_evidence:
  - 購入者の目的は確定した注文品を約束した宛先へ配送してもらうこと
  - 注文contextの「商品」は価格と購入数量を確定する注文品を意味する
  - 配送contextの「商品」は梱包単位と配送先を持つ配送品を意味する
  - 注文contextがOrderConfirmed eventを所有し、配送contextが下流consumerである
  - brokerはat-least-once deliveryで、同じeventの再配送が観測されている
  - 配送consumerの一時失敗後にeventが再配送される
  - 過去6か月に注文rule変更が配送処理の回帰を3回起こし、手動再受付と出荷遅延が発生した
  - roadmapでは注文ruleを毎月変更する予定が確認されている
  - 現在は注文と配送のruleが一つの共有modelへ混在している
known_unknowns:
  - duplicate eventの業務上の結果、deduplication owner、保持期間
  - retry上限、dead-letter後のrecovery owner、許容復旧時間
  - 段階改善のremediation costとpriority owner
allowed_assumptions:
  - artifact内の安定IDとHigh / Medium / Lowによる比較ラベル
prohibited_changes:
  - 根拠のない数値scoreでpriorityを確定する
  - transport設定から業務上のretryまたはduplicate resultを確定する
  - context別meaningを一つの共通定義へ統合する
  - unknownのownerまたはpolicyを一般知識で補う
  - repositoryを変更する
```
