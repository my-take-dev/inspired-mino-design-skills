## C1. Reservation confirmation and partial failure

```yaml
case_id: C1
skills: [$mino-reproducible-development]
mode: design
required_platforms: []
raw_request: >
  予約確定ユースケースのImplementation Specを作ってください。codeは変更しません。
confirmed_evidence:
  - actorは予約を購入する顧客と、失敗取引を照合する運用担当者
  - quantityは1以上8以下、moneyはamountとJPY currencyを失わない
  - stateはpending、confirmed、consumed、cancelled、expired
  - pendingからconfirmed/cancelled/expiredへ遷移できる
  - confirmedからconsumed/cancelledへ遷移でき、consumed後のcancelは禁止
  - 同じidempotency keyと同じrequestのduplicate confirmは同じ結果を返し、二重authorizationしない
  - 同じkeyでrequest内容が違う場合はconflictにする
  - payment authorization成功後にDB saveが失敗する可能性がある
  - writerはREST API、admin batch、migration importer
  - readerはcheckout API、invoice batch、support console
  - payment providerは一つで、追加予定のEvidenceはない
  - provider SDK型を公開contractへ出してはいけない
unknowns:
  - authorization成功後にsave失敗した場合、void、refund、reconciliationのどれを採用するか
  - recoveryの業務ownerと許容時間
allowed_assumptions:
  - artifact内の安定IDと説明用の名前
prohibited_changes:
  - recovery policyを一般知識から確定する
  - provider追加用Strategyまたはfactoryを作る
  - test planを実行済みとする
  - repositoryを変更する
```

## C2. Supporting audit-log migration

```yaml
case_id: C2
skills: [$mino-architecture-quality-strategy]
mode: design
required_platforms: [windows, linux]
raw_request: >
  法定監査logの保存基盤を段階移行するArchitecture Strategy Packageを作ってください。
confirmed_evidence:
  - 監査logは競争上の独自価値ではなくsupporting high-risk capability
  - 価値は法令順守、改ざん防止、障害時の追跡可能性を維持すること
  - Windows nodeとLinux nodeがlocal fileへ追記し、nightly importerがold databaseへ保存する
  - old databaseが現在のsource of truth
  - candidate meansとしてmanaged object storageが提示されているが未選択
  - 7年保持、append-only、生成から15分以内に検索可能がconfirmed requirement
  - 破壊的cutoverは未承認
unknowns:
  - target source of truth、dual-write期間、conflict rule、recovery owner
allowed_assumptions:
  - option比較用の論理ID
prohibited_changes:
  - 架空のunique valueまたはROIを作る
  - managed object storageを目的化する
  - Windowsだけの結果からcross-platform parityをpassにする
  - migrationまたはcutoverを実行する
```


## C4. Solution-first performance request

```yaml
case_id: C4
skills: [implicit-selection]
mode: design
required_platforms: []
raw_request: >
  checkout APIを速くするためRedisを導入したいです。実装やarchitecture選択はせず、
  まず何を解くべきか判断できる成果物を作ってください。
confirmed_evidence:
  - actorはcheckoutを完了する購入者
  - 500 requests/secondでcheckout APIのp95が2.4秒という直近計測がある
  - product ownerが同じ負荷でp95 1秒未満を成功条件として確認している
  - 公開APIのrequest、response、error semanticsは維持する
  - bottleneckと遅延原因は未調査
  - Redisはstakeholderが提示したcandidate meansで、採用決定ではない
allowed_assumptions:
  - artifact内の安定ID
prohibited_changes:
  - Redis採用またはcache方式を決定する
  - 遅延原因を推測で確定する
  - repositoryを変更する
  - architecture、contract、implementationを設計する
```
