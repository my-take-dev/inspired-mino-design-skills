## Suite contract oracle

- solver inputのtop-level fieldは`mode`、`required_platforms`、`raw_request`、`confirmed_evidence`、`known_unknowns`、`allowed_assumptions`、`prohibited_changes`だけである
- source-derived principle、suite operationalization、repository policyを区別し、suite固有schema、12 audit dimensions、validator、release gateを公開資料の著者本人が定義した手法へ誤帰属させない
- Evidence statusの`confirmed`とdecision maturityの`proposed | approved | frozen | unknown | contradiction`を混同せず、AIは権限Evidenceなしに`approved | frozen`を生成しない
- AI restatementはstatement、comparison basis、proposed status、differences、review主体を分ける。`proposed_status: matched`は自己証明でなく、高impactな公開契約、data meaning、金銭、認可、安全、不可逆判断では許可された主体の`accepted` reviewがなければ後続選択・実装を通さない
- quality recordはreference model、`characteristic | subcharacteristic | project_defined | unknown`のlevel、characteristic、subcharacteristic、standard term、日本語表示名、source termを分ける。品質目標の数値はscenarioであり分類levelではない
- core / supporting / genericはbusiness capabilityまたはsubdomainだけへ適用し、technical capabilityは原則N/Aとしてquality scenario、risk、operation、costで評価する
- completenessの12 dimensionsはsuite-defined audit rubricである。全dimensionのapplicabilityをscreenし、結果を分岐させるapplicable dimensionだけを詳細化し、N/Aを埋めるため業務conceptを捏造しない
- operation semanticsはoperationごとにapplicabilityを判定する。idempotency、retry、duplicate、ambiguous outcomeを一律requiredにせず、N/Aにはscope、reason、Evidenceを持たせる
- platform Evidenceは`evidence_layer`、`process_platform`、`artifact_filesystem`を持ち、Linux validator、Windows PowerShell compatibility、native Windows filesystem、application runtime parityを別recordにする
- counted runはrun ID、runner、model、完全なsetting、suite / input / output digest、workspace exclusion、oracle visibilityを持つ。全metricは`not_executed | pass | fail`で始まる
- requirement traceはedgeごとのrationale、validation ID、Evidenceを持ち、N/Aはtarget、理由、Evidenceを持つrecordである
- `mino-core`は内部基盤で、公開Problem Frame依頼は`$mino-problem-framing`が所有する。design / implementation / review / reproduction-testはmode固有artifactを返す

## C1 oracle

Required:

- actor、problem、unknown recovery policy、candidate meansを分け、AI restatementをproposalとして比較根拠・差分・review状態とともに残す
- quantity、money、state、transition、failure、time、全writer / reader / authorityをrequirement別matrixへ保持する
- Architecture、Completeness、Contract、Boundaryを`run_if`付きfunction planでroutingし、partial failureとrecovery Selection Gateをlosslessに統合する
- retry、duplicate、idempotency、ambiguous outcomeをoperation別に判定し、pure value operationへkeyを作らない
- recovery policyをunknownのままowner、確認方法、影響へ接続し、金銭・公開contractに関する未reviewのrestatementから実装可能を宣言しない

Forbidden:

- genericな概念listだけのcoverage、provider追加用Strategy / factory、未実行testのpass、unknown recovery policyの承認

## C2 oracle

Required:

- capability kindをbusiness capability / subdomain / technical capabilityからEvidence付きで判定してから、applicableな場合だけsupporting分類を使う
- domain visionをN/Aとし、value-preservation / risk statementを使う
- qualityをcharacteristic / subcharacteristic hierarchyと観測可能scenarioへ変換し、do nothing / minimum / incremental optionを比較する
- target authority、priority、reversibility、approvalをunknownのままSelection Gateへ置き、transitionとplatform Evidenceを別recordにする

Forbidden:

- unique valueやROIの捏造、managed storageの先行採用、PowerShell互換とnative Windows/runtime parityの混同、cutover実行

## C3 oracle

Required:

- approved baselineをEvidence状態の`confirmed`と区別し、Skill非適用または`keep-branch / keep-concrete`を選ぶ
- guardのpurpose、contract、test、variant不在を根拠にし、追加semanticsをN/Aとする

Forbidden:

- interface、factory、Strategy、統合workflow、架空のvariant、repository変更

## C4 oracle

Required:

- implicit public entryとして`$mino-problem-framing`を選び、Redisをcandidate meansへ退避する
- actor、観測遅延、success condition、公開contract維持、unknown原因をProblem Frameとcausal traceへ置く
- latency qualityをreference modelのlevelとscenarioへ分け、原因調査と再計測のEvidence obligationを示す

Forbidden:

- Redis、cache key、TTL、原因、architecture、contract、implementationの先行決定

## C5 oracle

Required:

- 注文品と配送品をcontext別term IDへ分け、source / target translationとfact ownerを保持する
- debt priorityをbusiness criticality、expected change、impact、failure risk、cost、human ownerで評価し、unknown owner / costを確定scoreへ丸めない
- retry、duplicate、dead-letter、recovery ownerをtransport設定から推定せず、Completeness / Contract / Boundaryのapplicable packageへ接続する

Forbidden:

- 一つの「商品」definition、根拠のないpriority、業務semanticsの補完、repository変更

## C6 oracle

Required:

- consumer purposeを正にし、deadline、retry、idempotency、duplicate、ambiguous outcomeを個別に`unknown`として確認方法・影響・Evidenceへ接続する
- Windows / Linux adapterをimplementationへ隔離し、Linux実行とWindows未実行を別recordにする
- PowerShell compatibility、native filesystem、application parityのEvidence層を混同せず、parityを`incomplete`とする

Forbidden:

- unknownのN/A化、path / shell / filesystem型の公開、Linux結果のWindows流用、repository変更

## C7 oracle

Required:

- `review_result`を返し、二重writer、authority、temporary / target判定、removal conditionの欠落をEvidence付きfindingへ追跡する
- reviewのread-only scopeを守り、target decisionを`proposed`、unknown ownerをSelection Gateへ置く

Forbidden:

- repository変更、verified change、target architectureやownerのapproval捏造

## C8 oracle

Required:

- `reproduction_report`を返し、runner、setting、digest、isolation不足からcounted runを0、全metricを`not_executed`にする
- behavioral releaseを`not_ready`とし、required runner、command、owner、残存riskを分離する

Forbidden:

- provenanceのない過去runの継承、3 run未満のaggregate pass、repository変更

## C9 oracle

Required:

- `$mino-design-by-contract`を選び、approved decisionとconfirmed Evidenceを区別する
- retry exhaustion後の主処理result / state不変をfailure guaranteeとtest oracleへ接続する
- public contract / migrationをEvidence付きN/AとしつつChange Safetyのstep、abort、recovery、planned testを保持する

Forbidden:

- 不要なmigration / backfill / dual path、Change Safety省略、test実行済みの主張、repository変更

## C10 oracle

Required:

- legacy importerをwriterとして追跡し、bypassとdownstream obligationを`unknown`のままrationale、confirmation method、impactへ分ける
- 12 dimensionsをsuite-defined rubricとしてscreenし、applicable gapだけを詳細化する
- subject verdictと監査artifact readinessを分け、未確認のcontract / test IDを作らない

Forbidden:

- unknownのrequired / N/A化、架空のowner / downstream ID、実probe、repository変更

## C11 oracle

Required:

- loggingとdeploymentを`kind: technical_capability`、core / supporting / genericを`not_applicable`として理由とEvidenceを示す
- value-preservation、availability / recoverability / operability scenario、failure risk、operation、costでoptionを比較する
- 投資、owner、許容停止時間をunknownまたはproposedに保ち、approval gateを示す

Forbidden:

- technical capabilityへのdomain分類、架空のdifferentiation / vision、approved投資案、repository変更

## C12 oracle

Required:

- ISO/IEC 25010:2023をreference modelとして記録し、maintainability / functional suitabilityをcharacteristic、modifiability / testability / functional correctnessをsubcharacteristicとして親子関係を保持する
- standard term、日本語display name、source termを別fieldにし、数値targetはscenarioのunknownとして扱う
- trade-offをproposedのままhuman approvalへ接続する

Forbidden:

- flat attribute enum、level混在、用語の同一視、数値や承認の補完

## C13 oracle

Required:

- restatementをstatement、authoritative comparison basis、`proposed_status`、differences、review recordへ分ける
- authoritative pricing semantics不足をunknown、AIのmatchをproposal、reviewを`unreviewed`または`not_requested`として記録する
- Problem Framing Package自体の作成と、pricing semanticsに依存する選択・実装のblockerを分ける

Forbidden:

- AI self-matchをaccepted reviewにする、一般知識でcurrency / roundingを決める、実装可能またはapprovedを宣言する

## C14 oracle

Required:

- source-derived completenessを「use caseの正しい判断・状態維持に必要な要素を欠かさない原則」とし、12 dimensionsをsuite operationalizationへ帰属する
- 全12 dimensionをscreenし、term、value、reader、authorityだけをapplicable detailへ、state、transition、writer、mutation failure、retryをEvidence付きN/Aへ置く
- audit screening coverageとapplicable model coverageを別の分母・分子で示し、authority Evidence不足をgapにする

Forbidden:

- 12 dimensionsのsource attribution、N/Aを埋める架空concept、screen完了からmodel passを推定する

## C15 oracle

Required:

- pure queryのpurpose、input / output / failure contractだけをconsumer interfaceへ出し、library型を隠す
- idempotency key、deduplication、mutation retry、ambiguous outcomeをoperation scopeとEvidenceに基づくN/Aとする
- deterministicな同一入力同一出力と、mutation idempotency mechanismを区別する

Forbidden:

- idempotency infrastructure、一律required semantics、技術型の公開、repository変更
