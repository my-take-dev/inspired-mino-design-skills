## Suite contract oracle

- solver caseのtop-level fieldは`mode`、`required_platforms`、`raw_request`、`confirmed_evidence`、`known_unknowns`、`allowed_assumptions`、`prohibited_changes`だけである
- benchmark metricはすべて`not_executed | pass | fail`で始まり、positive / negative polarityのboolean既定値を使わない
- counted runはrun ID、runner、model、完全なsetting、suite / input / output digest、workspace exclusion、oracle visibilityを持つ
- platform resultはrequirement、contract、verification、oracleのIDを保持し、未実行recordはreason、runner、command、owner、Evidenceを別fieldに持つ。platform固有のN/Aは`target_kind`と`target_id_or_scope`で対象を識別する
- canonical `decision.platform_validation`は`executed`と`unexecuted`を保持し、platform packageのrecordへ同じIDまたはEvidenceで接続する
- conditional designの未決選択はcanonical Selection Gateへ保持し、reversibilityその他のunknownを可逆または確定値へ丸めない
- standalone Function packageもPlatform Context / ValidationとapplicableなChange Safetyを保持する
- requirement traceはedgeごとのrationale、validation ID、Evidenceを保持し、N/Aはtarget、理由、Evidenceを持つrecordである
- manifestはsuite version / ownerをexactly once、skillをuniqueかつpath separatorなしで要求し、versioned artifactのheadingとbenchmark参照をversionへ一致させる
- solver fieldはallowlist、front matterとagent metadataはexact structure、100行超referenceのContentsはerrorとして両validatorが拒否する
- bare filename、absolute path、parent traversal、SkillsRoot外skill、case-insensitive text extensionを両validatorが同じ意味で検査する
- Skill内部のruntime file参照は`skills/`から始まる論理pathへ解決する
- `mino-core`は内部基盤であり、公開Problem Frame依頼は`$mino-problem-framing`が所有する
- design / implementation / review / reproduction-testはそれぞれImplementation Spec / Verified Change / Review Result / Reproduction Reportを返す

## C1 oracle

Required:

- actor、problem、unknown recovery policy、candidate meansを分け、AI restatementとcausal traceを残す
- Architectureはauthority / recovery slice、Completeness、Contract、Boundaryを`run_if`付きfunction planでroutingする
- quantity、money、state、transition、failure、time、全writer / reader / authorityをrequirement別matrixへ保持する
- missing behavior / relationship / time / writer / readerを識別可能なgapへ置き、unknown statusをpresentへ丸めない
- destruction probeがwriterからpartial failure、状態・副作用、顧客 / 運用impactまで追う
- Domain obligationのretry / duplicateを独立kindとしてContract Packageの実在IDへ接続する
- condition単位contract ID、authority type / owner、defensive validation、test ID / oracleを接続する
- idempotencyはconfirmでrequired、値objectのpure operationへkeyを作らない
- transport backoff、operation retry、duplicate result、ambiguous outcome recovery ownerを別recordにする
- recovery policyはunknownのままcanonical Selection Gate、確認方法、影響、Evidenceを示す
- router最終成果物にfunction plan、Core causal chain / selection gate、applicableなFunction packageがlosslessに残る
- design artifactのready / plannedと、後続implementation / releaseのblockerを分ける

Forbidden:

- genericな概念listだけでcoverageを主張する
- operation全体のowner一つへ全条件を置く
- provider追加用Strategy / factory
- planをpassed / executedとする
- unknown recovery policyをconfirmedにする

## C2 oracle

Required:

- domain visionを`not_applicable`とし、value-preservation / risk statementを使う
- security、recoverability、modifiability等をportfolioと観測可能scenarioへ変換する
- do nothing / do minimum / incrementalを含むoptionとtrade-offを比較する
- value → scenario → current finding → option → target decision → transition phase → validationをIDでtraceする
- target source of truth / authorityを未決として扱い、canonical Selection Gateを示す
- priority、reversibility、irreversible point、approval、recovery strategyのunknownを確定enumへ丸めない
- transitionへowner、deploy order、exit、irreversible point、abort、recovery、old path removalを持たせる
- temporary pathがowner、introduced date、purpose、metric / log、removal condition、removal phaseを持つ
- Windows / Linux recordが同じrequirement ID、contract ID、verification ID、oracle refを保持し、未実行recordとparity incompleteを分ける
- Architecture Strategy PackageがPlatform Context / ValidationとSelection Gateを保持する
- design artifact readyとruntime validation未実行を分ける

Forbidden:

- unique value、ROI、計測値の捏造
- managed object storageの先行採用
- trace IDなしのplatform parity判定
- transitionなしのtargetだけの提案
- migration / cutoverの実行

## C3 oracle

Required:

- Skillの非適用または`keep-branch / keep-concrete`を選ぶ
- guardのpurpose、固定contract、既存testを根拠にする
- add / change variant scenarioを`not_applicable`とし理由とEvidenceを示す
- review modeを守り、必要なら現行test実行だけを提案する

Forbidden:

- interface、factory、Strategy、追加layer
- full integrated workflowまたはArchitectureへのrouting
- 将来変更を根拠なく仮定する

## C4 oracle

Required:

- public entryとして`$mino-problem-framing`を選び、内部`mino-core`や統合routerを公開成果物として返さない
- Redisを未選択のcandidate meansへ退避し、購入者、観測遅延、成功条件、公開契約維持をProblem Frameへ置く
- bottleneckと原因をunknownにし、既存performance問題なのでcausal chainを`required`にする
- 具体計測 → actorのcheckout完了 → latency quality → 調査obligation → 再計測を往復traceする
- 原因調査と候補比較に必要なEvidence、反証条件、rejection criteriaを示す
- framing artifactのreadinessとarchitecture / implementationの未着手・blockを分ける

Forbidden:

- Redis、cache key、TTL、invalidation方式の選択
- 根拠のないDB、network、serialization等の原因確定
- architecture / contract / implementation成果物の作成

## C5 oracle

Required:

- priority assessmentがbusiness criticality、expected change、debt impact、failure risk、remediation costを一度ずつ持つ
- debt impactに出荷遅延・手動再受付・回帰の影響説明、Evidence、比較理由、人間owner recordを持つ
- ownerとremediation costがunknownなので`priority: unknown`とし、確認方法、影響、Evidenceを残す
- 注文品と配送品に別のstable term IDを付け、ambiguous termが両IDを参照し、actor / purpose / ruleをcanonical term recordへ保持する
- translationがcanonicalなsource / target context IDとterm IDを個別fieldで持ち、全IDがpackage内で解決する
- relationshipがstable ID、upstream / downstream context ID、fact owner、integration、consistency、relationship自体のEvidenceを持つ
- failure owner、retry、duplicate、ambiguous outcomeを独立判定し、unknownの確認方法と影響を隠さない
- canonical Domain Discovery PackageがCompleteness Packageとrouter最終成果物へlosslessに保持される
- transport redeliveryを業務上のretry可否またはduplicate resultの確定根拠にしない
- design artifactのreadinessと、未決policyに依存するimplementation blockerを分ける

Forbidden:

- 根拠のない数値priority scoreまたはowner不明時の確定priority
- 一つの`商品`definitionまたはterm IDへの統合
- free textだけのambiguous meaning / translation
- retry、duplicate、recovery ownerの推測
- design modeでのrepository変更

## C6 oracle

Required:

- consumer purposeとconsumer semantic operation contractを正にし、domain contractの適用状態をEvidence付きで判定する
- deadline、retry、idempotency、duplicate、ambiguous outcomeを独立して`unknown`とし、各確認方法、影響、Evidenceを持つ
- duplicate semanticsをidempotency mechanismから推定しない
- interface operation、implementation item、transport implementation、leakage finding、change scenario、traceをstable IDで接続する
- Windows / Linux adapterをimplementation partへ隔離し、logical report IDと同じconsumer contractを保つ
- Linux executed recordとWindows unexecuted recordを分け、Windows recordにreason、required runner、planned commands、owner、Evidenceを持たせる
- canonical decisionの`platform_validation.executed / unexecuted`がpackageのplatform recordへ同じIDまたはEvidenceで接続される
- parityを`incomplete`とし、一方のplatform結果を他方へ流用しない
- Boundary PackageがPlatform Context / ValidationとChange Safetyの適用判定を保持する

Forbidden:

- unknownをnot_applicableへ変更する
- path separator、shell、filesystem型をinterfaceへ公開する
- Linux testだけでplatform-neutral verifiedとする
- repository変更または架空のWindows実行結果

## C7 oracle

Required:

- `mode_artifact.kind: review_result`のReview Resultを返し、Evidence付きfinding、要件影響、subject verdict、canonical decisionを含める
- function planを最終成果物へ残し、実行したFunctionを`completed`、未決で止めたFunctionを`blocked`等へ更新する
- 二重writer、authority、transition controls、removal conditionの欠落をfindingへ追跡する
- unknown ownerとtarget / temporary判定をSelection Gateまたはcanonical unknownとして残す
- reviewのread-only validationと未実行validationを区別する

Forbidden:

- repositoryを変更する
- `implementation_spec`または`verified_change`を主成果物にする
- target architecture、owner、migrationを承認済みとして確定する

## C8 oracle

Required:

- `mode_artifact.kind: reproduction_report`のReproduction Reportを返す
- runner、model setting、suite / input / output digest、fresh isolationが不足しているためcounted runを0とする
- benchmark resultと全未採点metricを`not_executed`とし、過去summaryをpass / failへ変換しない
- unexecuted validationへreason、required runner、planned command、owner、Evidenceを分離して残す
- behavioral releaseを`not ready`とし、残存riskと次の再実行条件を示す

Forbidden:

- provenanceのない過去runを現在versionへ継承する
- 3 run未満または非隔離runからvariance、aggregate pass、release passを捏造する
- repositoryを変更する

## C9 oracle

Required:

- 明示Invocationどおり`$mino-design-by-contract`を選び、統合routerへ過剰routingしない
- approvedなretry変更を`intentional-change`としてbaseline化し、canonical Change Safetyを同じContract Packageへ保持する
- retry exhaustion後に主処理のresultとstateを変えないことをfailure guaranteeとtest oracleへ接続する
- public API、event、schema、永続dataに差分がないEvidenceから、public contract changeとmigrationを非適用として扱う
- migrationが非適用でも、変更step、abort、rollback / recovery、planned testをChange Safetyから落とさない
- design artifactのready / plannedと未実行testを分ける

Forbidden:

- 既存挙動の変更だけを理由にmigration contract、backfill、dual path、cutoverを作る
- Change Safetyをmigrationと同一視して省略する
- test planを実行済みとする
- repositoryを変更する

## C10 oracle

Required:

- 明示Invocationどおり`$mino-domain-model-completeness`をreview modeで使い、本番dataとfixtureを変更しない
- legacy importerをwriter access pathとしてstable IDで記録し、通常APIとは別経路として追跡する
- 未確認のbypassを`observed_result: unknown`、defenseとgapをunknownとして保持する
- `obligations.applicability: unknown`で、一般的な未決理由を`rationale`、code / schema / testの確認手順を`confirmation_method`、checkoutとsupport判断への影響を`impact_if_unresolved`へ分離する
- unknownのままcontract / test obligation IDを捏造せず、確認後のowner選択gateへ接続する
- 対象の`subject_verdict: indeterminate`と、Evidence付き監査artifactのreadinessを分ける

Forbidden:

- unknown obligationを`required`または`not_applicable`へ丸める
- confirmation methodとimpactを`rationale`だけへ圧縮する
- 架空のcontract / test IDまたはownerを作る
- repositoryを変更する
